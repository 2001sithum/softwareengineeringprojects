Configuration Management

Managing Change in Evolving Software Systems

What is Configuration Management?

Managing Complexity and Change

As software systems evolve, new versions are constantly created to fix bugs, add functionality, or support different hardware/OS platforms. Configuration Management (CM) is the process of managing these evolving systems, especially when maintained by teams.

Why is CM Necessary?

  • At any one time, there may be several different versions of the software in use by customers.
  • It is a crucial part of the general quality management process and is essential for certifications like ISO 9000.
  • CM tools are used to store versions, build systems from components, and track releases to customers.

Key Term: Baseline

When a version of a software system is released to CM, it becomes a controlled system or a baseline. A baseline is a stable, formally reviewed version that serves as a starting point for further development.

Configuration Management Planning

The CM Plan

A CM plan is created to define the standards and procedures for managing a project's artifacts. All products of the software process—specifications, designs, code, test data, and user manuals—must be managed.

The CM Plan Defines:

  • What to manage: The types of documents and a unique naming scheme (e.g., a hierarchical system).
  • Who is responsible: Defines who is in charge of CM procedures and creating baselines.
  • Policies: Defines the rules for change control and version management.
  • Tools: Describes the CM tools to be used and the process for using them.
  • Database: Defines the structure of the configuration database used to record all CM information.

The Configuration Database

All CM information should be maintained in a central configuration database. This database stores metadata about system components, changes, and users.

It should be able to answer critical questions like:

  • Which customer has a particular system version?
  • What hardware platform is required for version X?
  • What other versions are affected by a change to component Y?
  • How many reported faults exist in version T?

Change Management

A Controlled Process

Software systems are subject to continual change requests from users, developers, and market forces. Change management is the process of keeping track of these changes and implementing them in a cost-effective way.

The Change Management Process:

  1. A user or developer requests a change by completing a Change Request Form.
  2. The request is analyzed for validity and impact.
  3. The cost of the change is assessed.
  4. The request is submitted to a Change Control Board (CCB).
  5. The CCB, an independent group, decides if the change is cost-effective from a strategic viewpoint.
  6. If approved, the change is implemented and tested.
  7. A new system version is created.

Version Management

Tracking Instances of the System

Version management is the process of keeping track of the different versions of software components or systems.

Key Terms:

  • Version: An instance of a system that is functionally distinct from other instances (e.g., adding a new feature creates a new version).
  • Variant: An instance that is functionally identical but designed for different hardware or software configurations (e.g., a Windows version vs. a Linux version).
  • Release: An instance of a system that is distributed to users outside the development team.

Version Identification Techniques:

  • Version Numbering: A simple scheme (e.g., V1.1, V1.2), but can be inflexible for complex, branching development.
  • Attribute-based Identification: A flexible scheme where versions are identified by a set of attributes (e.g., language=Java, platform=XP). Supports powerful queries.
  • Change-oriented Identification: Integrates versioning with the changes made to create them, using "change sets".

Release Management

Delivering Software to Users

Release management is concerned with planning when and how to issue a system version as a release to customers. This is an expensive process that must be carefully planned.

What's in a Release?

A release is more than just an executable program. It includes:

  • Executable code
  • Configuration files
  • Data files needed for operation
  • An installation program or script
  • Electronic and paper documentation

Release Problems:

Customers may not want a new release if it provides unwanted functionality or they are happy with their current system. Release management cannot assume that previous releases have been accepted by all customers.

System Building

Creating an Executable System

System building is the process of compiling and linking software components into an executable system. This process is almost always automated using tools that are driven by 'build scripts' (e.g., a Makefile).

Common System Building Problems:

  • Missing Components: Build instructions do not include all required components.
  • Incorrect Component Version: The system is built with the wrong version of a component, which may cause it to fail after delivery.
  • Incorrect Platform: The system is not built for the correct target OS or hardware configuration.
  • Tool Incompatibilities: The wrong version of a compiler or other tool is used, generating different code.

For Agile and DevOps practices, frequent system building (e.g., daily builds, continuous integration) is essential. This helps find component interaction problems early and encourages thorough unit testing because developers don't want to "break the build."
