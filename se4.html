Verification and Validation

COM 2308 Software Engineering

V&V Core Concepts

Verification

The process of checking that the software conforms to its specification.

The Core Question:

"Are we building the product right?"

Validation

The process of checking whether the software does what the user really requires.

The Core Question:

"Are we building the right product?"

The V&V Process and Goals

Verification & Validation (V&V) is a whole life-cycle process, applied at each stage of software development. It's not just about testing at the end.

Principal Objectives:

  • The discovery of defects in a system.
  • The assessment of whether the system is useful and usable in an operational situation.

The Ultimate Goal:

To establish confidence that the software is fit for purpose. This does NOT mean completely free of defects. The software must be "good enough" for its intended use, and the level of confidence required depends on factors like system criticality, user expectations, and market pressures.

Static Verification

Analysis of static system representations (documents, code) to discover problems. The code is NOT executed.

  • Techniques: Software inspections, code reviews, automated static analysis.

Dynamic Verification

Observing the behavior of the product by executing it on a computer. The code IS executed.

  • Technique: Software testing with test data.

V&V Planning & The V-Model

The Importance of Planning

Careful planning is required to get the most out of testing and inspection. V&V planning should start early in the development process and define the balance between static and dynamic techniques. The output is a Test Plan, which defines standards for the testing process.

The V-Model of Development

The V-model illustrates how V&V activities can be integrated throughout the life cycle. It shows how test planning for each phase can happen during the design of that phase, linking the two sides of the "V".

  • Requirements Specification leads to the Acceptance Test Plan.
  • System Specification leads to the System Integration Test Plan.
  • System Design leads to the Sub-system Integration Test Plan.
  • Detailed Design leads to Module and Unit Tests.

Static Verification in Detail

1. Software Inspections

What are Inspections?

A formal process where people examine source code or documents to discover defects. They are intended for defect detection, not correction. They are highly effective because:

  • Many different defects can be found in a single session.
  • They aren't blocked by a single defect, unlike testing.
  • Reviewers can use their experience to spot common error types.

Inspections vs. Testing

They are complementary techniques, not opposing ones.

  • Inspections can check conformance with a specification (Verification).
  • Testing is needed to check conformance with real user requirements (Validation).
  • Inspections cannot check non-functional characteristics like performance or usability.

The Inspection Process & Roles

A formal inspection follows a defined procedure: Planning → Overview → Individual Preparation → Inspection Meeting → Rework → Follow-up.

RoleResponsibility
AuthorThe programmer or designer who created the work. Fixes discovered defects.
InspectorFinds errors, omissions, and inconsistencies.
ReaderPresents the code or document during the meeting.
ScribeRecords the results of the inspection meeting.
ModeratorManages the process and facilitates the inspection.

2. Automated Static Analysis

Software tools that scan source code to discover potentially erroneous conditions. They are a supplement to, not a replacement for, inspections.

Stages of Static Analysis:

  • Control Flow Analysis: Finds unreachable code, loops with multiple exits.
  • Data Use Analysis: Detects uninitialized variables or variables declared but never used.
  • Interface Analysis: Checks for consistency in function calls (e.g., correct number/type of parameters).
  • Information Flow Analysis: Identifies dependencies between input and output variables.
  • Path Analysis: Identifies all possible execution paths through the code.

Static analysis is particularly valuable for languages with weak type checking like C, as the compiler won't catch many common errors.

Dynamic Verification: Program Testing

The Nature of Testing

Testing involves executing a program with test data to observe its behavior. It is a fundamental part of V&V.

  • Testing can reveal the presence of errors, NOT their absence.
  • It is the only way to validate non-functional requirements (e.g., performance, reliability).
  • It should always be used in conjunction with static verification for full coverage.

Component (Unit) Testing

Testing individual program components (functions, classes) in isolation. This is typically the responsibility of the component developer.

System Testing

Testing groups of components integrated to create a system or sub-system. This is typically the responsibility of an independent test team.

Integration Testing

Building a system from its components and testing for problems caused by their interactions. Strategies include top-down and bottom-up integration.

Release Testing

Testing a complete version of the system that will be distributed to customers. This is usually "black-box" testing, performed without knowledge of the internal code structure.

Testing vs. Debugging

These are two distinct processes:

  • Testing (V&V): Is concerned with establishing the existence of defects.
  • Debugging: Is concerned with locating and repairing these defects.

Test Case Design

Test case design involves creating the specific inputs and expected outputs used to test the system. The goal is to create a set of tests that are effective at both validation (meeting requirements) and defect testing (finding bugs).

Design Approaches:

1. Requirements-based Testing

A validation technique where you derive at least one test case for every system requirement to ensure it is met.

2. Partition Testing

A black-box technique where you identify "equivalence partitions"—classes of input data where the program should behave equivalently (e.g., positive numbers, negative numbers, strings). Test cases are chosen from each partition and at their boundaries.

3. Structural (White-Box) Testing

A technique that uses knowledge of the program's internal structure to derive test cases. The objective is to exercise all program statements and execution paths. Path testing is a key example of this approach.

Test Automation & SQA

Test Automation

Testing is an expensive and time-consuming process. Testing workbenches provide a range of tools to help automate it and reduce costs. A workbench might include:

  • Test Manager: Manages the execution of tests.
  • Test Data Generator: Generates test data for the program.
  • Oracle: Generates predictions of expected test results.
  • File Comparator: Compares actual results with expected results.
  • Dynamic Analyzer: Instruments code to measure test coverage.
  • Simulator: Simulates parts of the system that are not yet available.

Software Quality Assurance (SQA)

SQA covers a broad set of activities to ensure a project is completed based on its specifications, without defects. It is about defining and managing the quality *process*.

Key SQA Responsibilities:

  • Audit and review all product deliverables.
  • Ensure test plans are executed correctly.
  • Deal with new or modified standards and practices.
  • Ensure quality control over subcontracted or outsourced products.
  • Collect process and product quality metrics.
