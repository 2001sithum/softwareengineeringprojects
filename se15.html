Question 1

a) Describe how Agile methods accommodate changing requirements.

(05 marks)

Answer & Explanation

Agile methods are fundamentally designed to embrace and accommodate change. They achieve this through several core practices:

  • Iterative Development: Work is broken down into small, time-boxed iterations (e.g., Sprints in Scrum). At the end of each iteration, stakeholders review a working increment of the software and can provide feedback, which allows requirements to be adjusted for the next iteration.
  • Prioritized Backlog: Requirements are maintained in a prioritized list (Product Backlog). This list is dynamic and can be re-prioritized at the start of each new iteration, allowing new or changed requirements to be moved up in priority.
  • Customer Collaboration: Agile values constant communication with the customer. An on-site customer representative provides immediate clarification and makes decisions about changing requirements, preventing delays.
  • Embracing Change as a Principle: The Agile Manifesto explicitly states "Responding to change over following a plan." This mindset encourages teams to expect and welcome changes rather than resisting them.
  • Simple Design & Refactoring: Agile teams avoid "Big Design Upfront" (BDUF). They create the simplest design that works for the current requirements and continuously refactor (improve the internal structure of) the code to make it easier to modify as requirements evolve.

b) Explain why cowboy coding is considered suitable for problems that are small and well understood.

(04 marks)

Answer & Explanation

Cowboy coding, an unstructured approach where developers (often a single programmer) implement solutions without a formal process, is suitable for small, well-understood problems because:

  • Low Communication Overhead: For a single developer or a very small, co-located team, the need for formal documentation, meetings, and planning is minimal. The developer has a complete mental model of the simple problem.
  • Speed of Delivery: It allows for extremely rapid development of a solution. For a small, urgent problem, the overhead of a formal process would be counterproductive and slow down delivery.
  • Prototyping/Spikes: It is ideal for creating quick prototypes or "spikes" to test a technical idea or explore a solution. The goal is learning, not building a production-ready system.
  • Minimal Risk: Because the problem is small and well-understood, the risks of failure, budget overruns, or creating an unmaintainable system are very low.

However, this approach fails completely for large, complex, or poorly understood problems where structure, planning, and collaboration are essential.

d) The following is a simple Python function that calculates the factorial of a number. Identify and explain the code quality issues present in the function.

def factorial(x):
    # this function calculates factorial
    result = 1
    i = 1
    while i <= x:
        result = result * i
        i = i + 1
    return result
(06 marks)

Answer & Explanation

While the function is logically correct for positive integers, it has several code quality issues related to readability, robustness, and standard Python practices:

  1. Poor Variable Naming: The parameter name x is not descriptive. A name like n or number would be clearer and more conventional for a mathematical function.
  2. Lack of Proper Documentation: The single-line comment is minimal. A proper Python docstring should be used to explain the function's purpose, its parameters, what it returns, and any exceptions it might raise.
  3. No Input Validation: The function does not handle edge cases or invalid inputs. For example:
    • If a negative number is passed, the loop condition i <= x will be immediately false, and it will incorrectly return 1.
    • If a non-integer (like a float or string) is passed, it will raise a TypeError at runtime.
  4. Un-Pythonic Loop: Using a while loop with a manual counter (i) is less "Pythonic" than using a for loop with range(), which is generally more readable and less prone to off-by-one errors.

Refactored Code Example:

def calculate_factorial(n: int) -> int:
    """Calculates the factorial of a non-negative integer.

    Args:
        n: A non-negative integer.

    Returns:
        The factorial of n.

    Raises:
        ValueError: If n is a negative integer.
        TypeError: If n is not an integer.
    """
    if not isinstance(n, int):
        raise TypeError("Input must be an integer.")
    if n < 0:
        raise ValueError("Factorial is not defined for negative numbers.")

    factorial_result = 1
    for i in range(1, n + 1):
        factorial_result *= i

    return factorial_result

Question 3

a) The following is a Python function to check if a `username` string meets certain criteria (must be between 5 and 15 characters). Using partition testing, explain the different classes of numbers you would consider as inputs to test this function. Assume `username` is an integer.

def validate_username(username: str) -> bool:
    length = len(username)
    return 5 <= length <= 15
(05 marks)

Answer & Explanation

Partition Testing (or Equivalence Partitioning) is a black-box testing technique where input data is divided into partitions of equivalent data from which test cases can be derived. All inputs in a partition are expected to be processed in the same way.

For the condition 5 <= length <= 15, we can identify three main partitions:

  1. Valid Partition (length is between 5 and 15 inclusive):
    • Description: This class includes all usernames that should return True.
    • Example Test Cases: A username with length 5 ("abcde"), length 10 ("abcdefghij"), and length 15.
  2. Invalid Partition 1 (length < 5):
    • Description: This class includes all usernames that are too short and should return False.
    • Example Test Cases: A username with length 4 ("abcd") and an empty string with length 0 ("").
  3. Invalid Partition 2 (length > 15):
    • Description: This class includes all usernames that are too long and should return False.
    • Example Test Cases: A username with length 16 ("1234567890123456").

Additionally, using Boundary Value Analysis (BVA), we would specifically test the values at and around the boundaries (5 and 15):

  • Length 4 (below the lower boundary)
  • Length 5 (on the lower boundary)
  • Length 15 (on the upper boundary)
  • Length 16 (above the upper boundary)
