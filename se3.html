Software Implementation

COM 2308 Software Engineering

The Implementation Phase

Software implementation is the process of translating a software design into executable code. It's the phase where the architectural and detailed designs become a tangible product.

  • It begins after the software design is finalized, reviewed, and approved.
  • Ideally, this is a straightforward mapping of design to code if the design is clear and unambiguous.
  • However, many software errors are introduced during this phase due to poor implementation practices.

Choosing a Programming Language

The selection of a programming language (PL) is a critical first step and depends on several factors:

  • Nature of the application: (e.g., web, embedded system, data science).
  • Cost: Licensing fees for compilers, IDEs, or runtimes.
  • Available resources: Expertise of the current development team.
  • Client expectations: Pre-existing technology stacks or performance requirements.

This choice can also impact HR activities, potentially requiring recruitment of new developers or training for the existing team.

Coding Style & Documentation

The Importance of Standards

A coding standard is a set of guidelines for a specific programming language that recommends programming style, practices, and methods. Adopting a standard is crucial for long-term project success.

  • Standards should be realistic, easy to follow, and practical.
  • Adherence is checked by Software Quality Assurance (SQA) and during code reviews.
  • There's a short-term cost in learning and applying standards, but it leads to massive long-term savings due to improved understandability and maintainability.

What can be standardized?

Naming Standards: Consistent naming for variables, functions, classes, etc.
Commenting Standards: Rules for how and what to comment.
Formatting Standards: Guidelines for layout, indentation, and spacing.

Example Naming Conventions:

int temp_i; // snake_case with type suffix for integer
float avgTempDuringDay_f; // camelCase with type suffix for float
const STUDENT_TABLE; // SCREAMING_SNAKE_CASE for constants/tables
int classGrades_ia; // Suffix for integer array

Software Reuse & APIs

Principles of Software Reuse

Leveraging existing code is a cornerstone of efficient software development.

  • Reusing trusted, well-tested components (libraries, OTS components) increases software reliability.
  • When writing new code, it should be designed with future reuse in mind.
  • This means writing code with a clear Application Programming Interface (API).

What is a Good API?

An API is the contract that allows different software components to communicate. A good API is a valuable corporate asset.

  • Easy to learn and use.
  • Secure and hard to misuse.
  • Easy to extend and adapt.
  • Stable over a long period of time.
  • Does one function well and hides private implementation details (encapsulation).

API Documentation Template

Proper documentation is essential for an API's usability. Here is a standard template for what should be included:

API Name 
Brief Description 
Syntax or Prototype (Header) 
Typical Use Scenario 
Usage Example 
Arguments Details 
Exceptions Thrown 
Returned Value 
Prerequisites / Preconditions 
Special Notes or Warnings 

Fault Tolerance & Exception Handling

Software Fault Tolerance

The ability of software to deal with errors and abnormal conditions, allowing it to detect, recover, and continue operation. Such software is also called error-recoverable, self-stabilizing, or resilient.

Approaches to Building Fault-Tolerant Software:

Checkpointing: Periodically save the 'state' of the software. On failure, recover from the last saved checkpoint (backward recovery) or move to a future desirable state (forward recovery).
N-Version Programming: N independent teams provide N implementations of a design. An arbiter takes a majority vote on the results. Fails if the design itself is wrong or the majority of implementations are faulty.
Recovery Block: Save the state, then try several versions of an operation in decreasing order of confidence. A 'judge' procedure accepts or rejects the result. If all fail, an error is invoked.

Exception Handling

A mechanism for dealing with run-time errors (exceptions) like division by zero or an invalid array index in a structured way.

The `try-catch-finally` Block

try {
    // Code that may cause or generate an exception
} catch (SpecificExceptionType ex) {
    // Code for handling this specific exception
} catch (AnotherExceptionType ex) {
    // Code for handling another type of exception
} finally {
    // Cleanup code that is executed in all cases,
    // whether an exception occurred or not.
}

Declaring Exceptions

A method that may throw a checked exception should declare it in its signature using the `throws` keyword.

public void readUserProfile(String fileName) throws IOException {
    if (fileName == null) {
        throw new IOException("File name cannot be null");
    }
    // ... more code
}

Writing Secure Software

Common Security Vulnerabilities

Security vulnerabilities are often introduced during implementation. Following secure coding practices and performing thorough code reviews can prevent or remove them.

I/O Validation & Representation Errors

  • Buffer Overflow: Writing data past the boundary of a buffer, overwriting adjacent memory.
  • Command Injection: Injecting malicious code (e.g., SQL, OS commands) through user input.
  • Denial of Service (DoS): Attacks that make a resource unavailable to legitimate users.

Security Violations

  • Improper or missing access control.
  • Use of weak random number generators.
  • Violation of the "least privilege" principle.
  • Improper password management.

API Abuse

  • When the caller of an API deviates from its contract.
  • Example: Ignoring a returned error value from a function, leading to unexpected behavior.

Coding Errors & Poor Quality

  • Dead Code: Code that is never executed.
  • Uninitialized Variables: Using a variable before a value has been assigned to it.
  • Memory Leaks: Failing to release memory that is no longer needed.

Code Review

Process and Purpose

A code review is a non-automatic, formal process where code is examined by other team members to find defects, improve quality, and share knowledge.

  • Normally done prior to dynamic testing, after the code compiles without errors.
  • Includes code walkthroughs and inspections using checklists.
  • Software Quality Assurance (SQA) sets the rules for when to distribute code and who participates.

Typical Participants in a Code Review:

Author: The programmer who wrote the code.
Designer: The person responsible for the system design.
Tester: A quality assurance team member.
SQA Representative: Ensures standards are met.
Review Leader: Moderates the meeting (often the SQA rep).
Review Recorder: Documents findings and action items.
