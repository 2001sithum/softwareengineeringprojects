Software Evolution

ICT 2402 Software Engineering

The Nature of Software Change

Why Software Must Change

Software change is inevitable. Once deployed, software must evolve to remain useful. The development process doesn't end at delivery; it continues in a spiral of evolution through multiple releases.

Key Drivers for Change:

  • New Requirements: Emerge as users interact with the software.
  • Business Environment: The business context changes, requiring new features or processes.
  • Error Repair: Bugs and faults must be fixed.
  • New Equipment: The software must adapt to new hardware or operating systems.
  • Performance Improvement: The system may need to be faster or more reliable.

A key problem for organizations is managing this constant change to their existing software systems effectively.

Program Evolution Dynamics

Lehman's Laws

Program evolution dynamics is the study of system change. Based on empirical studies, Lehman and Belady proposed a set of "laws" that are sensible observations about how large, tailored software systems evolve over time.

Law Description
Continuing Change A program used in the real world must change or become progressively less useful.
Increasing Complexity As a program evolves, its structure degrades, and complexity increases. Resources must be devoted to preserving and simplifying the structure (refactoring).
Large Program Evolution Evolution is a self-regulating process. System attributes like size and time between releases are roughly constant for each release.
Continuing Growth Functionality must be continually added to maintain user satisfaction.
Organisational Stability Over a program's lifetime, its rate of development is approximately constant.
Conservation of Familiarity The incremental change in each release is approximately constant, as radical changes can alienate users.

Software Maintenance

Types of Maintenance Activities

Maintenance is the process of modifying a program after it has been put into use. It usually involves modifying existing components and adding new ones, rather than major architectural changes.

Corrective

Fixing errors (bugs) found by users or testers after deployment. Often unscheduled and urgent.

Adaptive

Adapting the software to new operating environments, hardware, or technologies.

Perfective

Improving functional and non-functional requirements, such as performance or usability, based on user feedback.

Preventive

Improving the software's structure to enhance future maintainability and reduce future errors (e.g., refactoring).

Maintenance Cost Factors

Predicting and controlling maintenance costs is a major challenge. Key factors include:

  • Team Stability: Costs are lower if the original developers maintain the system.
  • Staff Skills: Maintenance staff are often less experienced, which can increase costs.
  • Program Age & Structure: As programs age, their structure degrades (increasing complexity), making them harder and more expensive to change.
  • Contractual Responsibility: If the original developers have no maintenance contract, they have no incentive to design for future change.

The Evolution Process

The system evolution process is driven by change proposals. It is a continuous loop that lasts for the entire system lifetime.

The General Process:

  1. Change Requests: A proposal for a change is made.
  2. Impact Analysis: The team analyzes the request to understand its effect on the system, the effort required, and the cost.
  3. Release Planning: If the change is accepted, it is planned for a future release. This may involve bundling several changes together.
  4. Change Implementation: The change is designed, coded, and tested. This follows a mini-SDLC: requirements analysis -> requirements updating -> software development.
  5. System Release: A new version of the system is deployed.

Urgent Changes (Emergency Repair)

Sometimes, urgent changes (e.g., fixing a critical system fault) must be implemented without going through the full formal process. This "quick fix" approach can degrade the system structure and must be followed up with proper re-engineering later.

Legacy System Evolution

Evolution Strategies

Organizations must choose a strategy for evolving their critical legacy systems. The choice depends on the system's business value and its technical quality.

Low Quality, Low Business Value

Strategy: Scrap it. These systems are a drain on resources and should be decommissioned.

Low Quality, High Business Value

Strategy: Re-engineer or Replace. These systems are critical but expensive to maintain. Improving their quality through re-engineering or replacing them is essential.

High Quality, Low Business Value

Strategy: Replace with COTS, Scrap, or Maintain. If a commercial off-the-shelf system can do the job, use it. Otherwise, scrap it or perform minimal maintenance.

High Quality, High Business Value

Strategy: Continue Normal Maintenance. These are valuable assets. Continue evolving them through the standard maintenance process.

System Re-engineering

Re-structuring or re-writing part or all of a legacy system without changing its functionality. The goal is to improve its structure and documentation to make it easier to maintain.

Advantages:

  • Reduced Risk: Avoids the high risks associated with new software development (e.g., staffing, specification problems).
  • Reduced Cost: The cost is often significantly less than developing a brand-new system.

Configuration Management

Managing Evolving Systems

As software evolves, new versions are created for different machines, with different functionality, or for specific users. Software Configuration Management (SCM) is the process of managing these evolving systems.

Key Goals of SCM:

  • Develop and apply procedures to manage a changing software product.
  • Control the costs and effort involved in making changes.
  • Manage system change as a team activity, ensuring that changes made by different developers don't conflict.

A released version of a software system under configuration management is often called a baseline, as it serves as a stable starting point for further development.
