Software Process Models

COM 2308 Software Engineering

Core Concepts

Learning Objectives

  • Categorize different software process models.
  • Describe three generic process models and their use cases.
  • Understand and discuss process iteration.
  • Briefly describe key process activities.

The Software Process

A structured set of activities required to develop a software system. The fundamental activities are:

  • Specification: Defining what the system should do.
  • Design & Implementation: Defining the organization of the system and implementing it.
  • Validation: Checking that it does what the customer wants.
  • Evolution (Maintenance): Changing the system in response to changing needs.

What is a Software Process Model?

According to Sommerville: "A software process model is an abstract representation of a process. It presents a description of a process from some particular perspective."

In simple terms, it's a simplified roadmap or strategy for how to move through the software process activities from start to finish.

Three Generic Models We Will Cover:

  1. The Waterfall Model: A sequential, plan-driven approach.
  2. Iterative Processes: An evolutionary approach emphasizing feedback and cycles.
  3. Component-based Software Engineering: A reuse-oriented approach.

1. The Waterfall Model

Concept: Sequential Flow

The Waterfall model treats the fundamental process activities as separate, distinct phases. The output of one phase is the input to the next, flowing downwards like a waterfall. There is no going back to a previous phase once it's complete.

Requirements Analysis
↓
System & Software Design
↓
Implementation & Unit Testing
↓
Integration & System Testing
↓
Operation & Maintenance

Advantages

  • Enforces a disciplined, structured approach.
  • Documentation is produced at each stage, improving project visibility and knowledge transfer.
  • It is more economical if bugs are found early in the process.
  • Ideal for creating highly reliable systems where requirements are stable.

Criticism & Disadvantages

  • Accommodating change is difficult and costly. It does not respond well to changing customer requirements.
  • It's often impossible to know all requirements and limitations at the start of a project.
  • The model implies you can perfect one phase before moving to the next, which is unrealistic.
  • A working version of the software is not available until late in the project lifecycle.
  • It does not explicitly cover risk management.

When to Use the Waterfall Model

  • Large, complex projects where the system is developed at several sites and coordination is key.
  • Routine projects where the requirements are well-understood and unlikely to change (e.g., building a system similar to one built before).
  • When the development organization is very familiar with the problem domain.
  • Critical systems (e.g., for NASA, Dept. of Defense) where extensive analysis and design must precede any implementation.

2. Iterative Processes

Concept: Evolve and Adapt

Iterative models acknowledge that software requirements often change. Instead of a single linear pass, process activities are repeated in cycles. Specification, development, and validation are interleaved, and the software evolves over multiple versions.

Key Iterative Models

A. Incremental Development

Based on developing an initial implementation, showing it to users, and evolving it through several versions until the system is complete. Each version (increment) adds more functionality.

Advantages over Waterfall
  • Reduced cost to accommodate changing requirements.
  • Easier to get early and frequent customer feedback.
  • Rapid delivery of useful software, starting with the highest-priority features.
  • Customers gain value early and don't have to wait for the entire system.
  • Most important services get the most testing.
Disadvantages
  • Process is not highly visible; managers can find it hard to measure progress without regular deliverables.
  • System structure can degrade as new increments are added without refactoring.
  • Can be hard to identify common facilities needed by all increments early on.
  • Conflicts with procurement models that require a complete system specification upfront.

B. Spiral Development

Represents the process as a spiral, where each loop is a phase. It explicitly incorporates risk assessment and reduction at each stage, making it suitable for large, expensive, and complicated projects.

The Four Sectors of Each Spiral Loop:
  1. Objective Setting: Specific objectives for the phase are identified.
  2. Risk Assessment and Reduction: Key risks are assessed, and activities are put in place to reduce them (e.g., prototyping).
  3. Development and Validation: A development model (like Waterfall or Incremental) is chosen for this phase of development.
  4. Planning: The project is reviewed, and the next loop of the spiral is planned.

C. Agile Software Development

A specific type of iterative development that emphasizes flexibility, collaboration, and speed. It focuses on delivering working software in very short timeboxes.

Core Principles:
  • Software is developed in short iterations (typically 1-4 weeks).
  • Working software is the primary measure of progress.
  • Real-time, face-to-face communication is preferred over extensive documentation.
  • The team re-evaluates project priorities at the end of each iteration.

3. Reuse-Oriented Engineering

Concept: Build from Existing Parts

This approach, also known as Component-Based Software Engineering (CBSE), focuses on integrating existing components or Commercial-Off-The-Shelf (COTS) systems rather than developing everything from scratch. It relies on a large base of reusable software components.

Process Stages:
Component Analysis
→
Requirements Modification
→
System Design with Reuse
→
Development & Integration

Advantages

  • Reduces the amount of software that needs to be developed.
  • Reduces overall cost and risk.
  • Allows for faster delivery of software.

Disadvantages

  • Requirements may have to be compromised to fit available components.
  • The development team does not have full control over the evolution of third-party components (e.g., updates, bugs).

Related Methodologies & Techniques

Extreme Programming (XP)

An influential Agile method that stresses customer satisfaction and empowers developers to respond to change. It is built on five values: Communication, Simplicity, Feedback, Courage, and Respect.

Key Activities:

  • Listening: To customer needs.
  • Designing: For simplicity and maintainability.
  • Coding: Often done in pairs (Pair Programming).
  • Testing: Automated unit and acceptance tests are written before the code.

Test-Driven Development (TDD)

A technique where tests are written *before* the code. It is a method of designing software, not just testing it. The cycle is short and rapid.

The TDD Cycle:

  1. Add a test: Write a new automated test for a small piece of functionality.
  2. Run tests: See the new test fail (and all others pass).
  3. Write code: Write the minimum amount of code to make the new test pass.
  4. Run tests: See all tests succeed.
  5. Refactor: Clean up the code while keeping the tests green.

Prototyping

Involves building a mock-up or early version of the system to get user feedback. It helps users understand the system and make design decisions early. However, it has drawbacks:

  • Users may expect the polished prototype to be the final product, delivered quickly.
  • There's a risk of using "patched-together" prototype code in the final system.
  • Can lead to too much focus on UI design over core functionality.

Cowboy Coding

An unstructured approach without a defined method, where team members (often a single programmer) do whatever they feel is right. It is only suitable for very small, well-understood problems or for creating a "spike" (a small experiment to test an idea).

  • Lacks a clear scope or vision.
  • Tends to produce low-quality, buggy software.
  • Lacks version control.
  • Represents a huge risk to the business for any non-trivial project.
