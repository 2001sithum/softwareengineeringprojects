Software Project Management

ICT 2402 Software Engineering

The Role of Management

What is Software Project Management?

It is concerned with all activities involved in ensuring that software is delivered:

  • On time and on schedule.
  • In accordance with the requirements of the organization.

It's essential because software development is always subject to budget and schedule constraints.

Unique Challenges (Distinctions)

  • The product is intangible: Progress is hard to see and measure.
  • The product is uniquely flexible: Requirements can change easily, complicating the process.
  • The process is not standardized: Unlike traditional engineering, many projects are 'one-off'.

Management Activities

  • Proposal writing
  • Project planning and scheduling
  • Project costing
  • Project monitoring and reviews
  • Personnel selection and evaluation
  • Report writing and presentations

Project Planning

A Continuous Process

Project planning is the most time-consuming management activity. It's not a one-time event but a continuous process from the initial concept through to system delivery. Plans must be regularly revised as new information becomes available.

The Project Planning Process Loop:

Establish project constraints
Make initial assessments of project parameters
Define project milestones and deliverables
while (project is active) {
Draw up project schedule
Initiate activities
// Wait for progress
Review project progress
Revise estimates of parameters
Update the project schedule
Re-negotiate constraints and deliverables
if (problems arise) {
Initiate technical review and revision
}
}

The Project Plan

The main project plan sets out the resources, work breakdown, and schedule. It has a defined structure and is supported by several other types of plans.

Project Plan Structure

  • Introduction
  • Project Organisation
  • Risk Analysis
  • Hardware & Software Resources
  • Work Breakdown
  • Project Schedule
  • Monitoring & Reporting

Supporting Plan Types

  • Quality Plan
  • Validation Plan
  • Configuration Management Plan
  • Maintenance Plan
  • Staff Development Plan

Project Scheduling

Organizing the Work

Scheduling involves breaking the project into tasks and organizing them to make optimal use of the workforce and resources. The key goals are:

  • Split the project into tasks and estimate their time and resource needs.
  • Organize tasks to be done concurrently where possible.
  • Minimize task dependencies to avoid delays.

Scheduling Problems

  • Estimating the difficulty and cost of software development is inherently hard.
  • Productivity is not proportional to team size (Brooks's Law: adding people to a late project makes it later due to communication overhead).
  • The unexpected always happens, so contingency must be built into the plan.

Scheduling Tools: Charts and Networks

Graphical notations are used to illustrate the project schedule.

Activity Networks (PERT Charts)

These charts show the dependencies between tasks. They are excellent for identifying the critical path—the sequence of tasks that determines the shortest possible duration for the project. Any delay in a critical path task directly delays the entire project.

Bar Charts (Gantt Charts)

These charts show the project schedule against calendar time. Each task is represented by a bar, showing its start date, end date, and duration. They provide a clear visual timeline for the project and are excellent for tracking progress.

Risk Management

Identifying and Planning for Adversity

Risk management is concerned with identifying risks and drawing up plans to minimize their effect on a project. A risk is a probability that some adverse circumstance will occur.

Types of Risk:

Project Risks: Affect the schedule or resources (e.g., staff turnover, hardware unavailability).
Product Risks: Affect the quality or performance of the software (e.g., requirement changes, size underestimate).
Business Risks: Affect the organization developing the software (e.g., technology change, product competition).

The Risk Management Process

A structured approach to handling risks throughout the project.

  1. Risk Identification: Identify potential project, product, and business risks. (e.g., technology, people, estimation risks).
  2. Risk Analysis: Assess the likelihood (probability) and seriousness (effects) of each risk. Classify probability (e.g., low, moderate, high) and effect (e.g., tolerable, serious, catastrophic).
  3. Risk Planning: Develop strategies to manage each risk.
    • Avoidance Strategies: Reduce the probability that the risk will arise.
    • Minimisation Strategies: Reduce the impact of the risk if it occurs.
    • Contingency Plans: Have a plan ready to deal with the risk if it materializes.
  4. Risk Monitoring: Regularly assess identified risks to see if their probability or impact has changed. Discuss key risks at progress meetings.

Key Points

  • Good project management is essential for project success.
  • The intangible nature of software presents unique challenges for management.
  • Planning, estimating, and scheduling are the most significant and iterative management activities.
  • A milestone is a predictable state where progress is formally reported.
  • Graphical representations like Gantt charts and activity networks are crucial for scheduling.
  • Risk management is a proactive process to identify and plan for threats to the project.
