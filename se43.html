Part 1: Overview of Software Engineering

Section 1: Core Definitions

  1. 1. Define 'Software' according to Ian Sommerville.

    Computer programs, associated documentation, and configuration data required to make these programs operate correctly.

  2. 2. List the three essential components that constitute 'software'.

    Programs, Documentation, and Configuration Data.

  3. 3. Define 'Software Engineering'.

    An engineering discipline that is concerned with all aspects of software production from system specification through to system maintenance.

  4. 4. What does the term 'all aspects of software production' encompass?

    It includes not only the technical processes of development but also project management and the development of tools and methods to support software production.

  5. 5. What is a 'software process model'?

    A simplified representation or abstraction of a software process, viewed from a particular perspective.

  6. 6. What is the 'Software Development Life Cycle' (SDLC)?

    A series of steps, stages, or phases in which software is produced, typically including requirements, design, implementation, testing, and maintenance.

  7. 7. Explain the concept of a 'software engineering method'.

    A structured approach to developing high-quality software in a cost-effective way, consisting of system models, notations, rules, and process guidance.

  8. 8. What are CASE tools?

    Computer-Aided Software Engineering tools are software systems designed to support software process activities like analysis, design, implementation, and testing.

  9. 9. What does it mean for software engineering to be an 'engineering discipline'?

    It means engineers selectively apply theories, methods, and tools appropriate for the project, but also recognize that some systems require creative, informal approaches.

  10. 10. What is the role of 'configuration data' in software?

    Configuration data consists of settings and files needed for the program to operate correctly in a specific environment, separate from the core executable code.

Section 2: Software Categories & Types

  1. 11. Differentiate between 'generic' and 'bespoke' software.

    • Generic: Stand-alone products sold on the open market (e.g., MS Office).
    • Bespoke: Custom software developed for a specific client (e.g., a company's inventory system).

  2. 12. Provide two examples of generic software.

    Spreadsheet applications (Excel), database systems (MySQL), text editors (VS Code).

  3. 13. Provide two examples of bespoke software.

    Air traffic control systems, embedded systems in a specific car model, a custom business process automation tool.

  4. 14. Differentiate between 'system' and 'application' software.

    • System Software: Manages computer resources and supports other programs (e.g., OS, compilers, drivers).
    • Application Software: Performs a specific function directly for the end-user (e.g., word processor, web browser).

  5. 15. Is a Database Management System (DBMS) system software or application software? Explain.

    It is system software because it manages a core computer resource (data storage and retrieval) and provides services to other application programs.

  6. 16. List three examples of system software.

    Operating Systems (Windows, Linux), Network Management Software, Compilers/Debuggers.

  7. 17. List three distinct categories of application software.

    General business productivity (spreadsheets), Home use (games), Cross-industry (accounting software), Vertical market (hospital management system).

  8. 18. What is 'vertical market' application software? Provide an example.

    Software for a specific industry. Example: A system for managing patient records in healthcare, or a system for managing telecommunication network billing.

  9. 19. What are the three ways of creating new software systems?

    1. Develop new software programs from scratch. 2. Configure generic software to suit an organization. 3. Re-use existing software components.

  10. 20. Classify antivirus software and explain your reasoning.

    It is a type of 'Utilities Software,' which is a sub-category of Application Software. While it interacts with the system at a low level, its primary function is to provide a specific utility directly to the end-user.

Section 3: The Software Crisis

  1. 21. In what year was the term 'software engineering' first proposed as a response to the 'software crisis'?

    1968.

  2. 22. What was the fundamental cause of the software crisis?

    Computer hardware technology was advancing rapidly, allowing for the creation of much more complex software. However, the methods and practices for developing software had not kept up with this increase in complexity.

  3. 23. List three common problems that characterized projects during the software crisis.

    1. Projects were running over budget and over schedule. 2. The resulting software was of low quality. 3. The software often did not meet the user's requirements.

  4. 24. How did 'maintainability' suffer during the software crisis?

    Projects were often unmanageable, and the code was poorly structured and difficult to maintain.

  5. 25. What was the proposed "answer" to the software crisis?

    The introduction of 'software engineering'â€”a disciplined, structured, and systematic approach to software development, inspired by traditional engineering fields.

  6. 26. What does it mean for a software project to be "unmanageable"?

    It means there were no standard methods to plan, track, and control the project's progress, leading to chaos, schedule slips, and budget overruns.

  7. 27. Why did the quality of software suffer during this period?

    Without standard practices for design, testing, and quality assurance, software was often unreliable, inefficient, and full of errors (bugs).

  8. 28. What was the most extreme outcome for some projects during the software crisis?

    Some projects were never delivered at all, collapsing under their own complexity and lack of management.

  9. 29. How did the software crisis relate to user requirements?

    Without a disciplined process for gathering and managing requirements, the final software often failed to meet the actual needs of the users.

  10. 30. What was the relationship between hardware and software advancements at the time?

    Hardware advanced rapidly, creating the potential for large, complex systems. Software development practices lagged far behind, unable to reliably deliver on that potential.

Section 4: The Software Process & SDLC

  1. 31. List the four fundamental stages of the software process.

    1. Software Specification. 2. Software Development. 3. Software Validation. 4. Software Evolution.

  2. 32. What two activities occur during Software Specification?

    Analysis and Specification.

  3. 33. What three activities occur during Software Development?

    Design, Implementation, and Documentation.

  4. 34. What two activities occur during Software Validation?

    Testing and Debugging.

  5. 35. What is the main activity during Software Evolution?

    Maintenance.

  6. 36. List the five shared steps/phases of almost all Software Development Life Cycles (SDLCs).

    Requirements, Design, Implementation, Testing, and Maintenance.

  7. 37. What are the four main goals of each step/stage/phase in an SDLC?

    1. Mark out a clear set of steps to perform. 2. Produce a tangible item (e.g., a document, code). 3. Allow for review of work. 4. Specify actions for the next phase.

  8. 38. Name two examples of life cycle models (or process models).

    Waterfall model and Spiral model. (Iterative and Component-based are also correct).

  9. 39. Explain the difference between 'Structured Analysis' and 'Object-Oriented' methods.

    • Structured Analysis: Breaks a system down into a set of functional components (processes and data flows).
    • Object-Oriented: Models the system as a collection of interacting objects that represent real-world entities.

  10. 40. What is the purpose of the 'Requirements' phase in the SDLC?

    To define what the system should do, capturing the needs of the users and stakeholders.

Section 5: Engineering Disciplines (CS, SE, System Eng.)

  1. 41. What is the primary focus of Computer Science?

    Theory and fundamentals of computation, algorithms, data structures, and the principles of computing.

  2. 42. How does Software Engineering differ from Computer Science?

    Software Engineering is focused on the practical application of Computer Science principles, along with project management and engineering discipline, to produce reliable and maintainable software systems.

  3. 43. What is System Engineering?

    A broader engineering field concerned with the development and evolution of *all* system components, including hardware, software, policies, and processes.

  4. 44. How does Software Engineering relate to System Engineering?

    Software Engineering is a *part* of System Engineering. While System Engineering covers all components, Software Engineering focuses on the software components in greater depth.

  5. 45. List two components that System Engineering is concerned with that Software Engineering is not.

    Hardware development and Policy/process design.

  6. 46. What are 'ad hoc approaches' in the context of Software Engineering?

    Practical, case-by-case methods or solutions that are not necessarily derived from formal theory but are used to solve specific problems during software production.

  7. 47. Differentiate between 'Upper-CASE' and 'Lower-CASE' tools.

    • Upper-CASE: Support early phases of the SDLC, such as requirements analysis and design.
    • Lower-CASE: Support later phases, such as implementation, testing, and maintenance.

  8. 48. Provide an example of an Upper-CASE tool's function.

    A tool for creating UML diagrams or for managing and tracing requirements.

  9. 49. Provide an example of a Lower-CASE tool's function.

    A compiler, a debugger, or an automated testing framework.

  10. 50. Why is the distinction between these engineering fields important?

    It clarifies roles and responsibilities. A software engineer needs to understand CS theory but must focus on practical delivery, and they must work within the broader context set by system engineers on large projects.

Section 6: Software Attributes & Objectives

  1. 51. Define 'Maintainability' as a software attribute.

    The ability of the software to evolve to meet changing requirements, fix bugs, or adapt to new environments.

  2. 52. Define 'Dependability' as a software attribute.

    The software should be reliable, safe, and secure, and should not cause damage when the system fails.

  3. 53. Define 'Efficiency' as a software attribute.

    The software should not waste system resources such as memory (RAM) or processor cycles (CPU time).

  4. 54. Define 'Usability' as a software attribute.

    The ability for end-users to use the software easily and without hassle to achieve their goals.

  5. 55. List seven key objectives of software engineering.

    Maintainability, Correctness, Reusability, Testability, Reliability, Portability, and Adaptability.

  6. 56. What is 'Correctness'?

    The extent to which the software conforms to its specification and is free from faults.

  7. 57. What is 'Reusability'?

    The extent to which parts of the software can be used in other applications.

  8. 58. What is 'Testability'?

    The ease with which the software can be tested to ensure that it is correct and meets its requirements.

  9. 59. What is 'Portability'?

    The ease with which software can be transferred from one hardware or software environment to another.

  10. 60. What is 'Adaptability'?

    The ability of the software to be adapted to different environments or to accommodate changes in requirements without major restructuring.

Section 7: Key Challenges in Software Engineering

  1. 61. What is the 'Legacy Challenge'?

    The challenge of maintaining and evolving old, valuable systems that were developed long ago using outdated technologies, which are often critical to business operations.

  2. 62. What is the 'Heterogeneity Challenge'?

    The challenge of developing and maintaining systems that are distributed across networks and must operate on a diverse range of hardware platforms and operating systems.

  3. 63. What is the 'Delivery Challenge'?

    The challenge of delivering systems quickly enough to meet the rapidly changing demands of the business and market, without sacrificing quality.

  4. 64. Why is maintaining legacy systems often difficult?

    They may lack documentation, use obsolete programming languages, have a degraded structure, and the original developers may no longer be available.

  5. 65. How do modern architectures like microservices relate to the Heterogeneity Challenge?

    They embrace heterogeneity by allowing different services within a single application to be built with different technologies, but this also increases the complexity of maintaining the overall distributed system.

  6. 66. Which type of software process model is best suited to address the Delivery Challenge?

    Iterative and Agile models are best suited, as they focus on delivering working software in small, frequent increments, allowing for rapid delivery of value.

  7. 67. Is it always best to replace a legacy system?

    No. If the legacy system is stable and still provides significant business value, the cost and risk of replacing it may be much higher than the cost of continuing to maintain it.

  8. 68. How does cloud computing impact the Heterogeneity Challenge?

    It can both simplify and complicate it. It simplifies by abstracting away hardware differences, but complicates it by introducing complex distributed network environments.

  9. 69. What is the main trade-off in the Delivery Challenge?

    The primary trade-off is between speed of delivery, cost, and the quality/reliability of the software. Moving faster often increases cost or reduces quality.

  10. 70. How are these three challenges interconnected?

    A legacy system (Legacy Challenge) that needs to connect to modern services (Heterogeneity Challenge) often needs changes delivered quickly to stay competitive (Delivery Challenge). Addressing one often impacts the others.

Section 8: Professional & Ethical Responsibilities

  1. 71. Which two major professional bodies collaborated on the Software Engineering Code of Ethics?

    The IEEE-CS (Institute of Electrical and Electronics Engineers - Computer Society) and the ACM (Association for Computing Machinery).

  2. 72. How many guiding principles are there in the code of ethics?

    There are 8 guiding principles.

  3. 73. What is the 'Public' principle?

    Software engineers shall act consistently with the public interest.

  4. 74. Explain the principle related to 'Client and Employer'.

    Engineers shall act in a manner that is in the best interest of their client and employer, *consistent* with the public interest. This means public interest comes first.

  5. 75. What is the 'Product' principle?

    Engineers shall ensure that their products and related modifications meet the highest professional standards possible.

  6. 76. What is the 'Judgment' principle?

    Engineers shall maintain integrity and independence in their professional judgment.

  7. 77. What is the 'Management' principle?

    Software engineering managers and leaders shall subscribe to and promote an ethical approach to the management of software development and maintenance.

  8. 78. What is the 'Profession' principle?

    Engineers shall advance the integrity and reputation of the profession consistent with the public interest.

  9. 79. What is the 'Colleagues' principle?

    Engineers shall be fair to and supportive of their colleagues.

  10. 80. What is the 'Self' principle?

    Engineers shall participate in lifelong learning regarding the practice of their profession and shall promote an ethical approach.

  11. 81. According to the code, what should an engineer do if their judgment is overruled in a way that endangers the public?

    They must report the situation to higher management and potentially to the appropriate authorities, as the 'Public' principle takes precedence.

  12. 82. How does the 'Product' principle relate to software quality?

    It obligates engineers to strive for high quality, reliability, and security in the software they produce, beyond just meeting the minimum requirements.

  13. 83. Why is "lifelong learning" considered an ethical responsibility?

    Because technology changes rapidly, and engineers have an ethical duty to keep their skills and knowledge current to ensure they are building safe and effective software.

  14. 84. Give an example of a conflict between the 'Client' and 'Public' principles.

    A client might ask an engineer to cut corners on security testing to save money and meet a deadline, but doing so could endanger the public's data. The engineer is ethically bound to prioritize public safety.

  15. 85. What does "independence in professional judgment" mean?

    It means an engineer's technical decisions should be based on merit and professional standards, and not be improperly influenced by managers or clients.

  16. 86. What is the key point of the key points section about software engineering?

    Software Engineering is an engineering discipline concerned with all aspects of software production.

  17. 87. What is the key point about software?

    Software = programs + documentation + configuration data.

  18. 88. What is the key point about the software process?

    The software process involves specification, development, validation & evolution.

  19. 89. What are the key software attributes mentioned in the summary?

    Maintainability, dependability, efficiency, and usability.

  20. 90. What is an objective for completing this lesson, regarding software categories?

    To be able to describe different categories of software.

  21. 91. What is an objective regarding the scope of software engineering?

    To be able to describe the scope of software engineering, including its objectives.

  22. 92. How does the lecture define the scope of system engineering vs. software engineering?

    System engineering involves the development of ALL system components (software, hardware, policy), while software engineering focuses in greater depth on the software components.

  23. 93. According to the lecture, what is the relationship between the Waterfall and Spiral models?

    The lecture presents them as examples of software process models. Waterfall represents process activities as separate phases, while Spiral is a form of iterative development.

  24. 94. What is the main message about creating new software?

    It can be done by developing new programs, configuring generic software, or re-using existing software.

  25. 95. What is the primary role of documentation in the definition of software?

    It is an essential, associated part of the software, not just an afterthought. It includes user manuals and technical guides.

  26. 96. Why are ethics a topic in a software engineering overview?

    Because software engineers have responsibilities not just to their clients but also to their profession and society as a whole, and these responsibilities must guide their work.

  27. 97. What kind of approach does an "engineering discipline" suggest?

    A systematic and organized approach where theories, methods, and tools are applied selectively and appropriately, rather than a chaotic or purely creative process.

  28. 98. What is the connection between the "software crisis" and "project management"?

    A major cause of the software crisis was a lack of project management, leading to projects being over budget and behind schedule. Software engineering introduced project management as a key aspect of production.

Part 2: Software Process Models

Section 1: Core Concepts & Process Activities

  1. 1. What is 'The Software Process'?

    A structured set of activities required to develop a software system.

  2. 2. List the four fundamental activities of the software process.

    Specification, Design & Implementation, Validation, and Maintenance.

  3. 3. Define a 'Software Process Model' according to Sommerville.

    An abstract representation of a process that presents a description from a particular perspective.

  4. 4. What is the primary purpose of a process model?

    To provide a roadmap or framework for the activities, actions, and tasks required to build high-quality software.

  5. 5. Name the three generic software process models discussed in the lecture.

    1. The Waterfall Model. 2. Iterative Processes. 3. Component-based Software Engineering.

  6. 6. What activity is involved in the 'Specification' phase?

    Defining what the system should do, including requirements gathering and analysis.

  7. 7. What happens during the 'Validation' phase?

    Ensuring the software meets the customer's needs, primarily through testing and debugging.

  8. 8. What is 'Process Iteration'?

    The concept of repeating software process activities to cater for changing requirements, forming the basis of iterative and agile models.

  9. 9. What is the main driver for process iteration?

    The fact that the requirements of a software project tend to change over time.

  10. 10. What is another name for the 'Maintenance' phase?

    Software Evolution.

Section 2: The Waterfall Model

  1. 11. Describe the core concept of the Waterfall model.

    It is a sequential process where development flows steadily downwards through distinct phases (Requirements, Design, Implementation, etc.). Each phase must be fully completed and signed off before the next one begins.

  2. 12. List the five main phases of the Waterfall model as shown in the detailed diagram.

    1. Requirements analysis and definition. 2. System and software design. 3. Implementation and unit testing. 4. Integration and system testing. 5. Operation and maintenance.

  3. 13. What is the main argument for using the Waterfall model regarding bug detection?

    It is more economical if bugs are found earlier in the lifecycle (e.g., during the requirements or design phase), as the cost to fix them increases exponentially in later phases.

  4. 14. What is a major advantage of the Waterfall model concerning project visibility?

    Documents are produced at the end of each stage (e.g., requirements specification, design document), which improves the visibility and traceability of the project for management.

  5. 15. What is the primary criticism of the Waterfall model?

    It is very difficult and costly to accommodate changing customer requirements because it requires going back to earlier phases and re-doing significant work.

  6. 16. Why does the Waterfall model struggle with real-world projects?

    Because it assumes that requirements can be fully known and frozen at the beginning, which is rarely possible. It also assumes that each phase can be perfected before moving to the next.

  7. 17. For what types of organizations is the Waterfall model still widely used?

    Large organizations like the US Department of Defense and NASA, especially for large, critical systems where upfront planning and documentation are mandatory.

  8. 18. When is the Waterfall model considered most applicable?

    When requirements are very well understood and stable, and the problem domain is familiar to the development organization.

  9. 19. What key management activity is not explicitly covered by the pure Waterfall model?

    Risk management.

  10. 20. How does the Waterfall model contribute to "waste of resources"?

    A working version of the software is not available until late in the cycle. If the final product does not meet user needs, much of the effort spent in the preceding phases is wasted.

Section 3: Iterative & Incremental Development

  1. 21. What is the core idea of Incremental Development?

    To develop an initial implementation, expose it to user comment, and then evolve it through several versions (increments) until an adequate system is developed.

  2. 22. How does the handling of requirements differ between Waterfall and Incremental models?

    In Waterfall, requirements are specified upfront. In Incremental models, the specification is developed in conjunction with the software, evolving with each iteration.

  3. 23. What are three advantages of incremental development over the waterfall model?

    1. Reduced cost to accommodate changing requirements. 2. Easier to get customer feedback on working software. 3. More rapid delivery of useful functionality.

  4. 24. How does incremental development provide value to the customer earlier?

    The first increment typically satisfies the most critical requirements, allowing customers to use the software and gain value immediately rather than waiting for the entire system to be finished.

  5. 25. What is a major management disadvantage of incremental development?

    The process is not as visible as in the Waterfall model. Without the regular, formal documents produced at each phase, it can be harder for managers to measure progress.

  6. 26. What tends to happen to the system structure in incremental development if not managed carefully?

    The system structure tends to degrade as new increments are added. Regular change can corrupt the structure unless time and money are spent on refactoring to improve it.

  7. 27. Why can getting useful customer feedback be difficult in iterative development for a replacement system?

    Users of the old system often want all of its functionality from day one and are unwilling to experiment with an incomplete new system, making it hard to provide feedback on early increments.

  8. 28. How does incremental development conflict with traditional procurement models?

    Many organizations' contracts require a complete system specification upfront. In an incremental approach, a complete specification doesn't exist until the final increment is defined, requiring a new form of contract.

  9. 29. Name three types of iterative development models mentioned in the lecture.

    Any three of: Incremental Development, Spiral model, Agile software development, Extreme Programming (XP), Test-Driven Development (TDD).

  10. 30. What does it mean for specification, development, and validation to be "interleaved"?

    It means these activities are not done in separate, long phases. Instead, a small amount of specification is done, followed by development and validation for that piece, and this cycle repeats.

Section 4: The Spiral Model

  1. 31. What is the defining characteristic of the Spiral model?

    It is a risk-driven process model where risks are explicitly assessed and resolved in each loop of the spiral.

  2. 32. For what types of projects is the Spiral model particularly favored?

    Large, expensive, and complicated projects where risk management is critical.

  3. 33. List the four sectors (activities) in each loop of the spiral.

    1. Objective setting. 2. Risk assessment and reduction. 3. Development and validation. 4. Planning.

  4. 34. What happens in the 'Objective setting' sector of a spiral loop?

    Specific objectives for that phase of the project (that loop of the spiral) are identified.

  5. 35. What happens in the 'Risk assessment and reduction' sector?

    Key risks are identified and assessed, and activities (like prototyping) are put in place to reduce those risks.

  6. 36. What happens in the 'Development and validation' sector?

    A development model (which can be any of the generic models, like waterfall) is chosen for that phase, and the system is developed and validated.

  7. 37. What happens in the 'Planning' sector?

    The project is reviewed, and the next phase (the next loop of the spiral) is planned.

  8. 38. Does the Spiral model have fixed phases?

    No, it does not have fixed phases like the waterfall model. The loops in the spiral are chosen depending on what is required at that stage of the project.

  9. 39. How is the process represented visually in the Spiral model?

    As a spiral rather than a sequence of activities with backtracking. Each loop represents a phase in the process.

  10. 40. What is the main benefit of the Spiral model's approach?

    It integrates risk management directly into the lifecycle, allowing high-risk problems to be addressed early before they become too costly or threaten the project.

Section 5: Agile Methods & Principles

  1. 41. What is the core idea behind Agile software development?

    To develop software in short timeboxes called iterations (1-4 weeks), releasing mini-increments of working software at the end of each iteration to adapt to changing requirements.

  2. 42. What is the primary measure of progress in Agile?

    Working software.

  3. 43. What form of communication does Agile emphasize?

    Real-time communication, preferably face-to-face, over extensive written documents.

  4. 44. How does Agile differ from general iterative development?

    Agile uses much shorter iterations (weeks vs. months), emphasizes a highly collaborative team manner, and treats the iteration length as a strict, non-negotiable timebox.

  5. 45. How does Agile differ from the Waterfall model in its approach to features?

    Waterfall completes entire phases (all design, then all coding). Agile aims to produce completely developed and tested features within a single iteration, focusing on the smallest workable piece of functionality first.

  6. 46. List the six typical phases within an Agile iteration.

    Planning, requirements analysis, Design, Coding, Testing, and Documentation.

  7. 47. What happens at the end of each Agile iteration?

    A mini-increment of the software is released, and the team re-evaluates project priorities for the next iteration.

  8. 48. Why can prioritizing changes be difficult in Agile?

    It can be difficult where there are multiple stakeholders with conflicting priorities, making it hard to decide which features to work on next.

  9. 49. Why does 'maintaining simplicity' require extra work in Agile?

    Because as new features are added iteratively, the system's design can degrade. It requires continuous effort (refactoring) to keep the code simple and easy to change.

  10. 50. What is the main advantage of delivering in "mini-increments"?

    It allows for rapid feedback from the customer on a tangible, working piece of software, which helps to ensure the final product meets their needs.

Section 6: Extreme Programming (XP)

  1. 51. What makes Extreme Programming (XP) 'extreme'?

    XP takes good Agile practices and turns them up to an 'extreme' level. For example, if code reviews are good, XP does them continuously via pair programming. If testing is good, XP writes tests first (TDD).

  2. 52. How frequently are increments delivered in XP?

    Typically every 2 weeks.

  3. 53. List the five essential values/ways XP improves a project.

    Communication, Simplicity, Feedback, Courage, and Respect.

  4. 54. What are the four main activities in XP?

    Coding, Testing, Listening, and Designing.

  5. 55. For what types of projects is XP most applicable?

    Projects involving new technology where requirements are expected to change rapidly, or for research projects focused on domain knowledge.

  6. 56. How does XP handle customer engagement?

    It requires full-time engagement of the customer or a customer representative who is part of the development team.

  7. 57. What is the purpose of 'listening' as an XP activity?

    Programmers must listen to what the customer needs and provide feedback to help improve the customer's own understanding of the problem.

  8. 58. What is the goal of 'designing' in XP?

    To create a simple design structure that organizes the system's logic in a way that allows one part to be changed without affecting other parts.

  9. 59. What are the two types of tests used in XP testing?

    1. Unit Tests: Automated tests to test the code. 2. Acceptance Tests: Based on requirements given by the customer.

  10. 60. What is the main goal of XP's approach?

    To stress customer satisfaction by empowering developers to confidently respond to changing requirements, even late in the life cycle.

Section 7: TDD, Prototyping & Cowboy Coding

  1. 61. What is Test-Driven Development (TDD)?

    A software development technique where you repeatedly write a test case first, and then implement only the code necessary to pass that test.

  2. 62. Is TDD primarily a method of testing or designing software?

    It is primarily a method of *designing* software, not just testing it. Writing the test first forces the developer to think about the desired interface and behavior.

  3. 63. List the five steps of the TDD cycle.

    1. Add a test. 2. Run all tests and see the new one fail. 3. Write some code. 4. Run the automated tests and see them succeed. 5. Refactor code.

  4. 64. What is the main purpose of prototyping in software development?

    To help users get an idea of the system and make design decisions without waiting for the full system to be built. It is a tool for clarifying requirements and getting early feedback.

  5. 65. What is a major drawback of showing a polished prototype to users?

    Users may see the prototype and have difficulty understanding that the finished, fully-functional design will not be produced for some time. It sets unrealistic expectations.

  6. 66. What is the risk of using "patched-together" prototype code?

    There is a strong temptation to use the low-quality prototype code in the final system to save time, which leads to a poorly engineered and unmaintainable product.

  7. 67. Define 'Cowboy coding'.

    An unstructured approach without a defined method, where team members (often just one programmer) do whatever they feel is right to get the job done.

  8. 68. In what limited situations is cowboy coding advantageous?

    For quick solutions to small, well-understood problems, or for creating a 'spike' to test a programming idea before committing to it in a larger project.

  9. 69. List three major disadvantages of cowboy coding for any non-trivial project.

    1. Lacks a clear scope/vision. 2. Tends to produce poor quality/buggy software. 3. Lack of version control.

  10. 70. Why does cowboy coding present a "huge risk to the business"?

    Because it often results in an unmaintainable, undocumented system that is dependent on a single developer. If that developer leaves, the business is left with a critical system that no one understands.

Section 8: Reuse-Oriented & Component-Based Engineering

  1. 71. What is the core concept of Component-Based Software Engineering (CBSE)?

    That systems should be built by integrating existing components or Commercial-Off-The-Shelf (COTS) systems, rather than developing all software from scratch.

  2. 72. What is another name for CBSE?

    Reuse-oriented development.

  3. 73. List the four main process stages of reuse-oriented development.

    1. Component analysis. 2. Requirements modification. 3. System design with reuse. 4. Development and integration.

  4. 74. What happens during the 'Component analysis' stage?

    The developers search for existing components that could be used to implement the system requirements.

  5. 75. What happens during the 'Requirements modification' stage?

    The requirements are analyzed against the components found and are modified to reflect what the available components can do.

  6. 76. List three advantages of CBSE.

    1. Reduces the amount of software to be developed. 2. Reduces cost and risk. 3. Allows for faster delivery of software.

  7. 77. What is the main disadvantage of CBSE regarding requirements?

    Requirements often have to be compromised because there may not be suitable components to match the exact client needs.

  8. 78. What is the main disadvantage of CBSE regarding evolution?

    The developing organization does not have full control over the evolution of the third-party components. They are dependent on the third party for upgrades and bug fixes.

  9. 79. What does COTS stand for?

    Commercial-Off-The-Shelf.

  10. 80. What is the key dependency for a successful CBSE approach?

    It requires a large base of available, reliable, and well-documented reusable software components and an integrating framework.

Section 9: Scrum Development

  1. 81. What is Scrum named after?

    The game of Rugby, in which a group is responsible for picking up the ball and moving it forward together as a team.

  2. 82. Is Scrum a process for developing software only?

    No, it is an iterative, incremental process for developing *any* product or managing *any* work.

  3. 83. What are the three main phases of the Scrum methodology?

    Pregame, Game, and Postgame.

  4. 84. What two activities happen during the 'Pregame' phase?

    Planning and System Architecture/High-Level Design.

  5. 85. What is the main activity during the 'Game' phase?

    Development occurs in 'Sprints', which are periods of concurrent engineering (Analysis, Design, Develop).

  6. 86. How long does a typical Sprint last?

    The slide says 30 days, though modern Scrum often uses shorter sprints (1-4 weeks).

  7. 87. When does testing happen in a Sprint?

    Testing happens throughout the Sprint, concurrently with development.

  8. 88. What happens in the 'Postgame' phase?

    Closure, which includes delivering a functioning deliverable, getting sign-off, and starting the next Sprint.

  9. 89. List four roles mentioned in Scrum.

    Scrum Master, Developer, QA, and Documentation Member.

  10. 90. List five values of Scrum.

    Flexible deliverable, Flexible schedule, Small teams, Frequent reviews, Collaboration, Team Empowerment, and Adaptability.

  11. 91. What happens in the "Adjust" part of a Sprint?

    The team makes adjustments to their plan or process based on what they learned during the Sprint review and their own internal reflection.

  12. 92. What is the key deliverable at the end of a Sprint?

    A functioning, potentially shippable increment of the product.

  13. 93. What is the focus of Scrum in terms of implementation?

    Scrum focuses on the entire organization for its implementation to be a success, not just the development team.

  14. 94. What is the first step in the 'Planning and Architecture' part of Pregame?

    Identify the project.

  15. 95. How does Scrum handle functional requirements?

    By prioritizing them during the planning phase.

  16. 96. What does "Team Empowerment" mean as a Scrum value?

    The development team is self-organizing and is given the autonomy to decide how best to accomplish their work during a Sprint.

  17. 97. Why are "frequent reviews" a core value?

    They provide a regular feedback loop with stakeholders, ensuring the product stays aligned with their needs and allowing for course correction.

  18. 98. What does 'Wrap' refer to in the Game phase?

    Wrapping up the development activities to prepare the increment for review and delivery.

  19. 99. What is the relationship between Scrum and Agile?

    Agile is a set of values and principles, while Scrum is a framework for implementing those principles. Scrum is one of the most popular ways to be Agile.

  20. 100. What is the overall message of the "Software Process Models" presentation?

    That there is no single "best" process model. The choice of model (e.g., Waterfall, Iterative, Agile) depends on the nature of the project, the stability of requirements, the level of risk, and the organizational context. Modern development favors iterative and agile approaches to handle change.

Part 3: Implementation, Verification & Validation

Section 1: V&V and Implementation Fundamentals

  1. 1. What is the core question of 'Verification'?

    "Are we building the product right?" (i.e., does it conform to its specification?).

  2. 2. What is the core question of 'Validation'?

    "Are we building the right product?" (i.e., does it meet the user's real needs?).

  3. 3. When does the software implementation phase typically begin?

    After the software design has been finalized, reviewed, and approved.

  4. 4. What are the two principal objectives of the V&V process?

    1. The discovery of defects in a system. 2. The assessment of whether the system is useful and usable in an operational situation.

  5. 5. What does it mean for the V&V process to establish that software is "fit for purpose"?

    It means establishing confidence that the software is good enough for its intended use, which is different from proving it is completely free of defects.

  6. 6. Differentiate between 'static verification' and 'dynamic verification'.

    • Static: Analysis of system representations (code, documents) *without* executing the program. E.g., code inspections.
    • Dynamic: Exercising and observing the program's behavior by *executing* it with test data. E.g., software testing.

  7. 7. List three factors that determine the required level of V&V confidence.

    1. System's purpose (how critical it is). 2. User expectations. 3. Marketing environment (e.g., speed to market vs. perfection).

  8. 8. What is the purpose of V&V planning?

    To carefully plan the testing and inspection processes to get the most out of them, and to identify the right balance between static verification and dynamic testing.

  9. 9. What is the V-model of development?

    A model that illustrates how V&V activities can be integrated throughout the development life cycle, showing how test plans for each development stage are created in parallel.

  10. 10. What are the two main causes of software errors mentioned in the implementation lecture?

    1. Poor implementation (coding errors). 2. Poor testing that fails to detect those errors.

Section 2: Implementation Practices (Coding Standards, Reuse, APIs)

  1. 11. List four factors that influence the selection of a programming language.

    Nature of the application, Cost (licensing), Available resources (team skills), and Client expectations.

  2. 12. What is a 'coding standard' and why is it important?

    A set of guidelines for writing code in a specific language. It is important because it improves code understandability, which in turn improves long-term maintainability.

  3. 13. Name the three main areas that can be standardized in coding.

    Naming standards, Commenting standards, and Formatting standards (indentation, spacing).

  4. 14. Why is it crucial for comments to be updated during maintenance?

    Outdated comments are misleading and can cause future developers to misunderstand the code, leading to new bugs. An incorrect comment is often worse than no comment at all.

  5. 15. What is the primary benefit of reusing well-tested software components?

    It increases the reliability of the new software, as the reused components have already been proven to work correctly.

  6. 16. What is an Application Programming Interface (API)?

    An API is the key to reusability. It defines a contract or a set of rules for how a software component can be used by other components, hiding its internal implementation details.

  7. 17. List four characteristics of a good API.

    Easy to learn and use, secure and hard to misuse, stable over a long time, and easy to extend.

  8. 18. What is the purpose of an API documentation template?

    To ensure that all necessary information for using the API (e.g., name, syntax, parameters, return values, exceptions) is documented in a consistent and thorough manner.

  9. 19. Explain the trade-off between short-term cost and long-term saving for coding standards.

    Adhering to coding standards may take slightly more time upfront (a short-term cost), but it saves a huge amount of time and money in the long term by making the code easier to read, debug, and maintain.

  10. 20. What is a code review?

    A non-automatic, formal meeting where code is examined by peers to find defects, improve quality, and ensure adherence to standards.

  11. 21. List four typical participants in a code review.

    The original programmer (author), a designer, a tester, and a Software Quality Assurance (SQA) representative.

Section 3: Secure Coding & Fault Tolerance

  1. 22. Define 'software fault tolerance'.

    The ability of a software system to detect and recover from errors and abnormal conditions, allowing it to continue operation.

  2. 23. Describe the 'Checkpointing' approach to fault tolerance.

    Periodically saving the 'state' of the software. If a failure occurs, the system can be restored to the last saved checkpoint (backward recovery).

  3. 24. Describe the 'N-version programming' approach.

    N independent teams provide N implementations of the same design. The results are passed to an arbiter that takes a majority vote.

  4. 25. What is the primary weakness of N-version programming?

    It fails if the original design itself is incorrect or if the majority of the implementations are faulty.

  5. 26. Describe the 'Recovery block' approach.

    A valid state is saved, and then several different versions (implementations) of an operation are tried in sequence. A 'judge' procedure accepts the first successful result. If all versions fail, an error is invoked.

  6. 27. What is 'exception handling' used for?

    For dealing with simple run-time errors in a structured way, such as divide by zero, invalid array index, or null pointer exceptions.

  7. 28. What is the purpose of the `try`, `catch`, and `finally` blocks?

    • try: Contains code that may cause an exception.
    • catch: Contains the code to handle the exception if it occurs.
    • finally: Contains cleanup code that is *always* executed, whether an exception occurred or not.

  8. 29. What is a 'Buffer Overflow' vulnerability?

    A serious security error where writing data to a buffer overruns its boundary and overwrites adjacent memory, which can be exploited by attackers.

  9. 30. What is a 'Command Injection' attack?

    An attack where malicious code (e.g., SQL commands, shell commands) is injected into the system through improperly validated input data.

  10. 31. Explain 'API abuse' as a security vulnerability.

    When a caller's module deviates from the API's contract, for example, by ignoring a returned error value. This can lead to unexpected and insecure system behavior.

  11. 32. List three common coding errors that affect code quality and can be detected by static checking.

    Dead code, Unused variables, Uninitialized variables, Memory leak, Type mismatch.

Section 4: Static Verification (Inspections & Analysis)

  1. 33. What is a software inspection?

    A formal process where people examine source code or documents to discover anomalies and defects. They do not require the system to be executed.

  2. 34. Why are inspections considered effective for discovering program errors?

    Many different types of defects can be discovered in a single session. Unlike testing, one defect does not mask another, and reviewers can use their experience to spot common error patterns.

  3. 35. Are inspections and testing opposing techniques?

    No, they are complementary. Both should be used during the V&V process for comprehensive coverage.

  4. 36. What can inspections check that testing cannot, and vice-versa?

    Inspections can check conformance with a specification and standards, but cannot check non-functional characteristics like performance. Testing is required for non-functional characteristics and validation against real user needs.

  5. 37. What is the main purpose of a program inspection: defect detection or defect correction?

    Defect detection only. The correction is done by the author *after* the inspection meeting.

  6. 38. List three essential pre-conditions for a successful inspection.

    1. A precise specification must be available. 2. The code must be syntactically correct. 3. Management must not use inspections for staff appraisal.

  7. 39. List the key roles in an inspection team.

    Author, Inspector, Reader, Scribe, and Moderator.

  8. 40. What is the purpose of an inspection checklist?

    To guide the inspection by providing a list of common errors to look for, specific to the language and domain (e.g., check for uninitialized variables, array bounds errors).

  9. 41. What is automated static analysis?

    The use of software tools to parse source code and discover potentially erroneous conditions, acting as an aid to inspections.

  10. 42. Describe 'data use analysis' as a stage of static analysis.

    It detects potential errors related to variables, such as uninitialized variables, variables that are declared but never used, or variables written to twice without an intervening read.

  11. 43. Why are static analysis tools less cost-effective for languages like Java compared to C?

    Because Java has strong type checking, its compiler already detects many of the errors (like type mismatches) that a static analysis tool would need to find in a weakly-typed language like C.

  12. 44. What are 'formal methods' in the context of static verification?

    They are the "ultimate" static verification technique, involving detailed mathematical analysis of a system's specification to develop formal arguments that a program conforms to that specification. They are very expensive and require specialized skills.

Section 5: Dynamic Verification (Testing Fundamentals)

  1. 45. What is the fundamental truth about program testing regarding errors?

    It can reveal the *presence* of errors, but it can NOT prove their *absence*.

  2. 46. For which type of requirements is testing the only validation technique?

    Non-functional requirements (such as performance and reliability), because the software must be executed to see how it behaves under load.

  3. 47. Differentiate between 'defect testing' and 'validation testing'.

    • Defect Testing: Aims to discover system defects. A successful test is one that makes the system fail and reveals a bug.
    • Validation Testing: Aims to show that the software meets its requirements. A successful test shows that a requirement has been properly implemented.

  4. 48. What is the difference between 'testing' and 'debugging'?

    Testing is the process of finding the existence of defects. Debugging is the process of locating the source of those defects and repairing them.

  5. 49. What is the first step in the debugging process after a test fails?

    Locate the error (the specific part of the code causing the failure).

  6. 50. What is a 'test plan'?

    A document that describes the standards for the testing process, including the scope, approach, resources, and schedule of intended testing activities.

  7. 51. List four key sections of a software test plan.

    The testing process, Requirements traceability, Testing schedule, and Test recording procedures.

  8. 52. What is 'requirements traceability' in a test plan?

    The process of ensuring that every requirement in the specification has at least one corresponding test case designed to validate it.

  9. 53. What is the purpose of test recording procedures?

    To ensure the results of tests are systematically recorded so that the testing process can be audited and repeated.

  10. 54. Explain what a 'test case' is.

    A set of inputs, execution conditions, and expected outputs designed to test a particular aspect of a system.

Section 6: Testing Levels, Types & Planning

  1. 55. What is 'Component testing' (or Unit testing)?

    The process of testing individual program components (e.g., functions, classes) in isolation. It is usually the responsibility of the component's developer.

  2. 56. What is 'System testing'?

    The process of testing groups of components that have been integrated to create a system or sub-system. It is usually the responsibility of an independent testing team.

  3. 57. What are the two phases of system testing?

    1. Integration testing. 2. Release testing.

  4. 58. What is 'Integration testing'?

    Building a system from its components and testing it for problems caused by component interactions. It is often done incrementally (top-down or bottom-up).

  5. 59. What is 'Release testing'?

    The process of testing a release of a system that will be distributed to customers. Its primary goal is to increase stakeholder confidence.

  6. 60. Is release testing typically 'black-box' or 'white-box' testing? Explain.

    It is typically 'black-box' testing. This means the testers base their tests on the system specification only and do not have knowledge of, or access to, the internal source code.

  7. 61. What is 'Performance testing'?

    A type of testing that involves planning a series of tests where the load on the system is steadily increased until its performance becomes unacceptable.

  8. 62. What is 'Stress testing'?

    Exercising the system *beyond* its maximum design load to see how it behaves when it fails. It checks for catastrophic failures and unacceptable loss of service or data.

  9. 63. Why does inheritance make object class testing more difficult?

    Because a class inherits all the operations from its super-class(es). The test suite for the subclass must test all inherited operations, as well as any new or overriding operations.

  10. 64. Who is typically responsible for component testing?

    The developer who wrote the component.

  11. 65. Who is typically responsible for system testing?

    An independent testing team.

Section 7: Test Case Design & Automation

  1. 66. What is the main goal of test case design?

    To create a set of tests that are effective in both validation (checking requirements) and defect testing (finding faults) without being exhaustive.

  2. 67. List the three design approaches for test cases.

    1. Requirements-based testing. 2. Partition testing. 3. Structural testing.

  3. 68. What is 'Requirements-based testing'?

    A validation testing technique where you derive a set of tests for each individual requirement to ensure it has been implemented.

  4. 69. What is an 'equivalence partition'?

    A class or domain of input data where the program is expected to behave in an equivalent way for every member of that class (e.g., all positive numbers).

  5. 70. What is the core idea of partition testing?

    To choose test cases from each identified equivalence partition and at the boundaries between partitions, as this is an effective way to find errors.

  6. 71. What is 'Structural testing' also known as?

    White-box testing.

  7. 72. What is the objective of structural testing?

    To use knowledge of the program's internal structure (code) to derive test cases with the objective of exercising all program statements or paths.

  8. 73. What is 'Path testing'?

    A structural testing technique where the objective is to ensure that the set of test cases executes every independent path through the program at least once.

  9. 74. What is a 'testing workbench'?

    A collection of tools designed to reduce the time and cost of testing by supporting activities like automatic execution, test data generation, and results comparison.

  10. 75. What is a test 'oracle' in a testing workbench?

    A component that generates predictions of the expected test results, which can then be compared against the actual results.

  11. 76. What is the role of a 'file comparator' in a workbench?

    It compares the actual output of a program test with the expected output (from the oracle or previous test results) and reports any differences.

  12. 77. What is the purpose of a 'dynamic analyzer'?

    It adds code (instruments the program) to count the number of times each statement has been executed, helping to measure test coverage.

  13. 78. What is a 'test manager' tool used for?

    Managing the running of program tests, including scheduling, execution, and logging results.

  14. 79. What is Software Quality Assurance (SQA)?

    A broad set of activities that ensure a project is completed based on its agreed-upon specifications without defects. It is about managing the quality of the *process*.

  15. 80. List three activities of an SQA group.

    1. Training of testers. 2. Review of test plans. 3. Adherence to testing standards.

  16. 81. What is the relationship between SQA and testing?

    Testing is an activity to find defects in the *product*. SQA is a broader set of activities to ensure the quality of the entire *process*, which includes managing the testing activities.

  17. 82. What is a 'stub' in testing?

    A dummy piece of code used to stand in for a lower-level module that has not yet been developed. It is used in top-down integration testing.

  18. 83. What is a 'driver' in testing?

    A piece of code that calls a lower-level component being tested and passes it test data. It is used in bottom-up integration testing.

  19. 84. What is 'regression testing'?

    The process of re-running existing tests after a change has been made to the software to ensure that the change has not introduced new bugs or broken existing functionality.

  20. 85. Why is test automation crucial for regression testing?

    The number of regression tests grows with every new feature. Manually running them all after every change is impractical. Automation makes it feasible to run the full suite of tests frequently.

  21. 86. What is 'black-box' testing?

    A testing technique where the tester does not have any knowledge of the internal code structure. Tests are based solely on the requirements and specifications.

  22. 87. In which testing phase is black-box testing most common?

    System testing and Release testing.

  23. 88. In which testing phase is white-box testing most common?

    Component (Unit) testing, where the developer has full knowledge of the code.

  24. 89. What is the main benefit of top-down integration testing?

    It allows for early demonstration of a working, albeit skeletal, version of the system, which can be good for user feedback and morale.

  25. 90. What is the main benefit of bottom-up integration testing?

    It allows for thorough testing of critical, low-level components first before they are integrated into the larger system.

  26. 91. What does "test coverage" measure?

    The degree to which the source code of a program has been executed by a particular test suite. It can be measured in terms of statement coverage, branch coverage, or path coverage.

  27. 92. Is 100% test coverage sufficient to guarantee a bug-free program?

    No. 100% coverage means every line of code was executed, but it does not mean it was executed with every possible input or in every possible state. It also cannot find missing requirements.

  28. 93. What is the primary role of a 'debugger' tool?

    To help a programmer locate a bug by allowing them to execute code line-by-line, inspect the values of variables, and set breakpoints.

  29. 94. What is 'alpha' testing?

    A type of acceptance testing performed by users/testers *within* the development organization, but outside the development team.

  30. 95. What is 'beta' testing?

    A type of acceptance testing where a release is distributed to a number of existing customers who use it in their own environment and report any problems.

  31. 96. What is the primary purpose of acceptance testing?

    To enable the customer to determine whether or not to accept the system.

  32. 97. What is 'usability testing'?

    A form of testing that focuses on assessing how easy the software is to use by observing real users as they attempt to complete tasks.

  33. 98. What is a 'test harness'?

    A collection of software and test data configured to test a program unit by running it under varying conditions and monitoring its behavior and outputs. It includes stubs and drivers.

  34. 99. What is the primary benefit of having an independent testing team?

    An independent team is not subject to the same pressures as the development team and can be more objective. Developers can be psychologically resistant to finding errors in their own code.

  35. 100. What is the overall message of the "Verification & Validation" presentation?

    That V&V is a whole-lifecycle process critical for building confidence in software. It requires careful planning and a combination of complementary techniques, including static methods like inspections and dynamic methods like multi-level testing, to be effective.

Part 4: Advanced Process Models

Section 1: Rational Unified Process (RUP)

  1. 1. What is the Rational Unified Process (RUP)?

    A modern, iterative software development process model derived from work on the Unified Modeling Language (UML).

  2. 2. From which three perspectives is RUP normally described?

    1. A dynamic perspective (phases over time). 2. A static perspective (process activities/workflows). 3. A practice perspective (good practices).

  3. 3. List the four phases of the RUP dynamic perspective.

    Inception, Elaboration, Construction, and Transition.

  4. 4. What is the main goal of the 'Inception' phase in RUP?

    To establish the business case for the system and define its scope.

  5. 5. What is the main goal of the 'Elaboration' phase?

    To develop an understanding of the problem domain, establish the system architecture, and mitigate key risks.

  6. 6. What is the main goal of the 'Construction' phase?

    To perform the main development work: system design, programming, and testing to build the product.

  7. 7. What is the main goal of the 'Transition' phase?

    To deploy the system to the end-users' operating environment, including beta testing, training, and data migration.

  8. 8. What is a 'workflow' in the static perspective of RUP?

    A process activity or discipline that groups related tasks. Examples include Requirements, Analysis & Design, and Implementation.

  9. 9. Which workflow is concerned with modeling business processes using business use cases?

    Business Modeling.

  10. 10. Which workflow involves identifying actors and developing system use cases?

    Requirements.

  11. 11. Which workflow is a supporting workflow that manages changes to the system?

    Configuration and Change Management.

  12. 12. According to the RUP graph, in which phase is the effort on 'Requirements' typically highest?

    The Elaboration phase.

  13. 13. In which phase is the effort on 'Implementation' and 'Test' highest?

    The Construction phase.

  14. 14. In which phase is the effort on 'Deployment' highest?

    The Transition phase.

  15. 15. List three of the six 'good practices' suggested by the RUP practice perspective.

    Any three of: Develop software iteratively, Manage requirements, Use component-based architectures, Visually model software, Verify software quality, Control changes to software.

Section 2: Agile Methods & Principles

  1. 16. What led to the creation of Agile methods?

    Dissatisfaction with the overheads, extensive documentation, and rigidity involved in traditional heavyweight design methods like the Waterfall model.

  2. 17. What is the main focus of Agile methods compared to traditional methods?

    Agile methods focus on the code and producing working software rather than on extensive upfront design and documentation.

  3. 18. For what size and type of project are Agile methods probably best suited?

    Small to medium-sized business systems or PC products where requirements are expected to change.

  4. 19. Explain the Agile principle of 'Customer Involvement'.

    The customer should be closely involved throughout the development process to provide and prioritize requirements and to evaluate the iterations of the system.

  5. 20. Explain the Agile principle of 'Incremental Delivery'.

    The software is developed in small increments, with the customer specifying the requirements to be included in each increment.

  6. 21. Explain the Agile principle of 'People, Not Process'.

    The skills of the development team should be recognized and exploited. The team should be trusted and left to develop their own ways of working without overly prescriptive processes.

  7. 22. Explain the Agile principle of 'Embrace Change'.

    Expect that system requirements will change and design the system in a way that can accommodate these changes.

  8. 23. Explain the Agile principle of 'Maintain Simplicity'.

    Focus on simplicity in both the software being developed and the process used. Actively work to eliminate unnecessary complexity.

  9. 24. What is a common problem with the 'Customer Involvement' principle?

    It can be difficult to keep the interest and commitment of customers who are required to be involved in the process, as it is a significant time investment for them.

  10. 25. Why might team members struggle with Agile methods?

    Some team members may not be able to cope with the intensity of team involvement, collaboration, and rapid pace needed in agile methods.

  11. 26. What is a common problem with contracts in Agile development?

    Traditional fixed-price contracts are based on a fixed set of requirements, which conflicts with the Agile principle of embracing change. New, more flexible contract models are often needed.

  12. 27. What is the primary measure of progress in an Agile project?

    Working software.

Section 3: Extreme Programming (XP)

  1. 28. What is Extreme Programming (XP)?

    Perhaps the best-known and most widely used Agile method that takes certain good practices of software development to 'extreme' levels.

  2. 29. Give two examples of what makes XP 'extreme'.

    1. New versions may be built several times per day (extreme integration). 2. Increments are delivered to customers every 2 weeks (extreme small releases). 3. Full-time engagement of the customer (extreme customer involvement).

  3. 30. What is 'Incremental Planning' in XP?

    Requirements are recorded on 'Story Cards'. The customer prioritizes stories, and developers break them down into 'Tasks' for planning a release.

  4. 31. What is the 'Small Releases' principle in XP?

    The minimal useful set of functionality that provides business value is developed first. Releases are frequent (e.g., every 2 weeks) and incrementally add new functionality.

  5. 32. What is the 'Simple Design' principle in XP?

    Only enough design is carried out to meet the *current* requirements and no more. This avoids over-engineering for a future that may not happen.

  6. 33. What is 'Test-First Development' (TDD)?

    The practice of writing an automated unit test *before* the functionality is implemented. The code is then written with the goal of making that test pass.

  7. 34. What is 'Refactoring' in XP?

    The continuous process of improving the internal structure of the code without changing its external behavior. This keeps the code simple and maintainable.

  8. 35. What is 'Pair Programming'?

    Two developers work together at one workstation, with one 'driving' (writing code) and the other 'navigating' (reviewing and thinking strategically). It acts as a continuous code review.

  9. 36. What is 'Collective Ownership'?

    The principle that anyone on the team can change any part of the system's code. This prevents knowledge silos and encourages shared responsibility.

  10. 37. What is 'Continuous Integration'?

    As soon as a task is complete, the code is integrated into the whole system. After integration, all unit tests in the system must pass.

  11. 38. What is 'Sustainable Pace'?

    The practice of avoiding overtime, as it leads to burnout and reduced code quality. The team should work at a pace that can be sustained long-term.

  12. 39. What is the 'On-Site Customer'?

    A representative of the end-user who is available full-time to the XP team to answer questions, write stories, and provide immediate feedback.

  13. 40. What is a 'user story' in XP?

    A short, high-level description of a requirement written from the perspective of the user. They are written on cards and used for planning.

  14. 41. How does XP handle changes, as opposed to conventional wisdom?

    Conventional wisdom is to design for anticipated future changes. XP maintains this is not worthwhile as changes cannot be reliably anticipated. Instead, it proposes constant refactoring to make the code easy to change when the need arises.

  15. 42. What is the main benefit of Test-First Development?

    Writing tests before code clarifies the requirements to be implemented and ensures that all new functionality is covered by automated tests, preventing regression errors.

  16. 43. How does Pair Programming spread knowledge across the team?

    As pairs rotate and work on different parts of the system, every developer gains familiarity with the entire codebase, which supports collective code ownership.

Section 4: Scrum Development

  1. 44. What is Scrum?

    An iterative, incremental agile framework for developing any product or managing any work, named after the game of Rugby.

  2. 45. What are the three high-level phases of the Scrum methodology shown in the presentation?

    Pregame, Game, and Postgame.

  3. 46. What two activities occur in the 'Pregame' phase?

    Planning and System Architecture / High-Level Design.

  4. 47. What is the core activity of the 'Game' phase?

    Development occurs in 'Sprints', which are periods of concurrent engineering.

  5. 48. What is a 'Sprint'?

    A time-boxed iteration (e.g., 30 days) where analysis, design, development, and testing occur concurrently.

  6. 49. What is the outcome of the 'Postgame' phase?

    Closure: delivering a functioning deliverable, getting sign-off, and preparing to start the next Sprint.

  7. 50. List the four activities of the 'Planning and Architecture' stage in Pregame.

    1. Identify the project. 2. Prioritize functional requirements. 3. Identify available resources. 4. Establish the target environment.

  8. 51. When does testing occur during a Sprint?

    Testing happens throughout the Sprint, not as a separate phase at the end.

  9. 52. What are the "Review" and "Adjust" activities within a Sprint?

    Review refers to demonstrating the work to stakeholders (Sprint Review), and Adjust refers to the team reflecting and improving its process (Sprint Retrospective).

  10. 53. List three roles on a Scrum team as per the presentation.

    Scrum Master, Developer, QA, Documentation Member.

  11. 54. List four of the Scrum values mentioned.

    Flexible deliverable, Small teams, Frequent reviews, Collaboration, Team Empowerment, Adaptability.

  12. 55. What does "Team Empowerment" mean as a Scrum value?

    It means the development team is self-organizing and is given the autonomy and trust to figure out how to best accomplish their work during a Sprint.

  13. 56. Why is "Adaptability" a key value in Scrum?

    Because Scrum is built on the idea that requirements and priorities will change. The process is designed to adapt to these changes every Sprint.

Section 5: Process Improvement & CMMI

  1. 57. What is 'process improvement'?

    The activity of understanding existing processes and introducing changes to improve product quality, reduce costs, or accelerate schedules.

  2. 58. What are the three stages of the process improvement cycle?

    1. Measure the current process. 2. Analyse the measurements to identify weaknesses. 3. Change the process to address those weaknesses.

  3. 59. What is the purpose of 'Process Measurement'?

    To measure attributes of the current process to create a baseline for assessing whether future changes are actual improvements.

  4. 60. What is the purpose of 'Process Analysis'?

    To assess the current process and identify bottlenecks, weaknesses, and areas for improvement.

  5. 61. What does CMMI stand for?

    Capability Maturity Model Integration.

  6. 62. What is the CMMI framework?

    A framework for process assessment and improvement developed by the Software Engineering Institute (SEI).

  7. 63. List the three main components of the CMMI model.

    Process areas, Goals, and Practices.

  8. 64. In CMMI, what are 'Process Areas'?

    Clusters of related practices that, when performed collectively, achieve a set of goals considered important for improving process capability (e.g., Project Planning, Requirements Management).

  9. 65. What are 'Goals' in the CMMI model?

    Descriptions of desirable organizational states. Each process area has associated goals.

  10. 66. What are 'Practices' in the CMMI model?

    Advisory ways of achieving a goal. Other approaches may be used, but the practices describe a typical path to success.

  11. 67. What is the difference between the 'staged' and 'continuous' CMMI models?

    • Staged: Assesses an organization's overall maturity on a single 5-level scale. An organization must satisfy all process areas at a lower level before moving to the next.
    • Continuous: A finer-grain model that assesses maturity for each individual process area. This allows organizations to pick and choose which areas to improve based on their specific needs.

  12. 68. List the five maturity levels of the staged CMMI model.

    1. Initial, 2. Managed, 3. Defined, 4. Quantitatively Managed, 5. Optimizing.

  13. 69. What does it mean for a process to be at the 'Initial' level?

    The process is chaotic, ad-hoc, and unpredictable. Success depends on individual heroics.

  14. 70. What does it mean for a process to be at the 'Managed' level?

    Basic project management processes are established to track cost, schedule, and functionality.

  15. 71. What does it mean for a process to be at the 'Defined' level?

    The process is documented, standardized, and integrated into an organizational standard.

  16. 72. What does it mean for a process to be at the 'Quantitatively Managed' level?

    The organization uses statistical and other quantitative techniques to manage and control the process and product quality.

  17. 73. What does it mean for a process to be at the 'Optimizing' level?

    The organization focuses on continual process improvement through quantitative feedback and piloting of innovative new ideas.

  18. 74. What is a 'process capability profile'?

    A graphical representation from the continuous CMMI model that shows the organization's maturity level for each individual process area, highlighting strengths and weaknesses.

  19. 75. What is the primary focus of most process improvement work so far?

    Defect reduction, reflecting the industry's increasing focus on quality.

  20. 76. What is the main advantage of the continuous CMMI model?

    It allows organizations to prioritize and improve specific process areas according to their local needs and business goals, rather than following a rigid, one-size-fits-all path.

  21. 77. What is a 'generic goal' in CMMI?

    A goal that applies to multiple process areas, such as "The process is institutionalized as a defined process."

  22. 78. Name two process areas under the 'Project Management' category in CMMI.

    Project Planning, Project Monitoring and Control, Risk Management.

  23. 79. Name two process areas under the 'Engineering' category in CMMI.

    Requirements Management, Requirements Development, Technical Solution, Verification, Validation.

  24. 80. Name two process areas under the 'Support' category in CMMI.

    Configuration Management, Measurement and Analysis.

  25. 81. What is a CMMI assessment?

    An examination of the processes used in an organization to assess their maturity in each process area, based on a 6-point scale.

  26. 82. What does "Not Performed" mean in a CMMI assessment?

    It means the process area is either not performed or does not completely achieve its goals. It is equivalent to Level 0.

  27. 83. What is the relationship between RUP and Agile?

    Both are iterative process frameworks. RUP is often seen as more formal and heavyweight, while Agile methods like Scrum and XP are more lightweight and people-centric. A project can tailor RUP to be quite agile.

  28. 84. Why are "User Stories" used in XP instead of formal requirement documents?

    They are lightweight, easy to understand, and encourage conversation between developers and the customer. They represent a "promise for a conversation" rather than a rigid specification.

  29. 85. What is a 'task card' in XP?

    A card that breaks a user story down into smaller, technical implementation tasks that developers can use for scheduling and cost estimates.

  30. 86. Can a project use both Scrum and XP practices?

    Yes. Scrum is a management framework, while XP provides a set of engineering practices. Many teams use Scrum for their overall process and adopt XP practices like TDD and Pair Programming to improve their development work.

  31. 87. What is the 'Environment' workflow in RUP?

    A supporting workflow concerned with making appropriate software tools (like IDEs, version control, etc.) available to the software development team.

  32. 88. How does the CMMI model support business goals?

    By providing a structured path for organizations to improve their processes, which can lead to higher product quality, lower costs, and more predictable schedules, all of which support business objectives.

  33. 89. Why is 'Visually model software' a good practice in RUP?

    Using a standard notation like UML helps to manage complexity, improve communication among team members, and create a clearer understanding of the system's architecture and design.

  34. 90. What is the core similarity between all advanced process models discussed?

    They are all fundamentally iterative and incremental, rejecting the rigid sequential approach of the pure Waterfall model.

  35. 91. What is the purpose of a task breakdown in XP?

    To break down user stories into small, manageable implementation tasks that form the basis for schedule and cost estimates.

  36. 92. What is the main philosophy of agile methods regarding design?

    To focus on creating a simple design that meets current needs and to evolve it through refactoring, rather than performing extensive upfront design for future, uncertain requirements.

  37. 93. According to the Scrum methodology diagram, what are the sub-activities of the 'Develop' step in a Sprint?

    Analysis, Design, and Develop. This shows that these activities are done concurrently within the Sprint.

  38. 94. What is a key challenge of prioritizing changes when multiple stakeholders are involved?

    Different stakeholders often have conflicting priorities, which can lead to disagreements and delays in deciding what the team should work on next. This requires a strong Product Owner to manage.

  39. 95. What does the "iteration" bar at the bottom of the RUP dynamic/static perspectives graph represent?

    It shows that each phase (e.g., Elaboration, Construction) is itself composed of one or more smaller iterations, reinforcing the iterative nature of the process.

  40. 96. What is the relationship between RUP and UML?

    RUP was derived from the work on UML and its associated processes. One of the good practices of RUP is to "Visually model software," typically using UML.

  41. 97. What is a common way requirements are documented in XP?

    As scenarios or user stories written on physical or digital cards.

  42. 98. What is the primary output of a Scrum Sprint?

    A potentially releasable, "Done" increment of the product.

  43. 99. What is the main benefit of the process improvement cycle (Measure, Analyse, Change)?

    It provides a data-driven, systematic approach to improving processes, ensuring that changes are based on evidence and their effects can be measured.

  44. 100. What is the overall message of the "Advanced Process Models" presentation?

    That modern software development has moved towards iterative, agile, and structured frameworks like RUP, XP, and Scrum to better handle complexity and changing requirements. Furthermore, continuous process improvement, often guided by models like CMMI, is essential for organizational maturity.

Part 5: Configuration Management

Section 1: Configuration Management Fundamentals

  1. 1. What is Configuration Management (CM)?

    The process of managing evolving software systems, especially when maintained by teams. It involves tracking and controlling changes to all software artifacts.

  2. 2. List three reasons why new versions of software systems are created.

    1. To support different machines/OS. 2. To offer different functionality. 3. To fix bugs.

  3. 3. What is a 'baseline' in CM?

    A stable, formally approved version of a software system or component that serves as a starting point for further development.

  4. 4. How does CM relate to quality management processes like ISO 9000?

    The definition and use of CM standards for traceability and control are essential for achieving ISO 9000 certification. CM is considered a key part of a general quality management process.

  5. 5. What should CM standards define?

    They should define how configuration items are identified, how changes are controlled, and how new versions are managed.

  6. 6. Why are new CM standards needed for Agile Development?

    Because traditional standards are often based on a Waterfall process. Agile relies on concurrent development and testing, requiring standards that support frequent system building and integration.

  7. 7. Describe the process of concurrent development and testing in an Agile context.

    A time is agreed for delivery of components. A new system version is built from these components. This new version is delivered for testing with pre-defined tests. Faults are documented and returned to developers.

  8. 8. What are the two main advantages of frequent system building?

    1. It is easier to find problems from component interactions early. 2. It encourages thorough unit testing, as developers are under pressure not to 'break the build'.

  9. 9. What is required to support frequent system building?

    A stringent change management process to keep track of discovered and repaired problems.

  10. 10. What is a 'configuration item'?

    Any product of the software process that is managed under CM control, such as a specification, design document, source code file, or test data.

Section 2: CM Planning & The CM Plan

  1. 11. Why is CM planning necessary for large projects?

    Because large projects can generate thousands of separate documents (specifications, designs, code, manuals) that must all be managed in a controlled and consistent way.

  2. 12. What is a CM plan?

    A document that defines the standards, procedures, tools, and responsibilities for configuration management on a specific project.

  3. 13. List four key things that a CM plan defines.

    1. The types of documents to be managed and their naming scheme. 2. Who is responsible for CM procedures. 3. Policies for change control and version management. 4. The CM records that must be maintained.

  4. 14. What information about tools should be included in the CM plan?

    The plan should describe the tools to be used, any limitations on their use, and the defined process for using them.

  5. 15. Why is a unique identification scheme for configuration items important?

    To avoid ambiguity. With thousands of documents, a unique identifier ensures that everyone is referring to and working with the correct item and version.

  6. 16. What is the most flexible approach for a document naming scheme?

    A hierarchical scheme with multi-level names, so that related documents have related names (e.g., `PCL-TOOLS/EDIT/FORMS/DISPLAY/CODE`).

  7. 17. What is a configuration database (or CMDB)?

    A database where all CM information is maintained. It stores metadata about components, users, changes, faults, and execution platforms.

  8. 18. Give two examples of queries a configuration database should be able to answer.

    1. Which customer has a particular system version? 2. What versions are affected by a change to component X?

  9. 19. How should a CM database ideally be linked to the software being managed?

    It should be part of an integrated development environment, where the database and the managed documents are all maintained on the same system.

  10. 20. What is a common practical implementation for a CM database?

    Often, the files themselves are maintained in a Version Control System (like Git or CVS), while the metadata about those files is stored in a separate CM database, as fully integrated CASE tools can be expensive.

Section 3: Change Management

  1. 21. What is the primary concern of change management?

    Keeping track of continual change requests and ensuring that they are implemented in the most cost-effective way.

  2. 22. List three sources of change requests.

    From users, from developers, and from market forces.

  3. 23. What is the first step in the change management process?

    Requesting a change by completing a change request form.

  4. 24. What is the role of the Change Control Board (CCB)?

    An external group that reviews change requests to decide if they are cost-effective from a strategic and organizational viewpoint, rather than a purely technical one. They approve or reject changes.

  5. 25. Why should the CCB be independent of the project team?

    To provide an objective assessment of whether a change is beneficial for the organization as a whole, rather than just being technically interesting or convenient for the project team.

  6. 26. What information is recorded on a change request form?

    It records the proposed change, the requestor, the reason, urgency, impact analysis, cost estimate, and the final decision/recommendations.

  7. 27. What is the purpose of change tracking tools?

    To keep track of the status of each change request and automatically route the requests to the right people at the right time (e.g., via email integration).

  8. 28. What is a 'derivation history'?

    A record of changes applied to a specific document or code component. It typically includes the change made, the rationale, who made it, and when.

  9. 29. Where is the derivation history often stored?

    It is often included as a comment header in the source code file itself.

  10. 30. What is the 'if change is accepted then...' loop in the change process?

    It is the implementation loop: make changes to software, submit for quality approval, and repeat this process until the quality is adequate, at which point a new system version is created.

Section 4: Version Management

  1. 31. What are the four key activities of version management?

    1. Invent an identification scheme. 2. Plan when new versions are produced. 3. Ensure procedures and tools are properly applied. 4. Plan and distribute new system releases.

  2. 32. Define 'Version'.

    An instance of a system that is functionally distinct in some way from other system instances (e.g., V1.1 has a new feature that V1.0 does not).

  3. 33. Define 'Variant'.

    An instance of a system that is functionally identical but is designed for different hardware or software configurations (e.g., a Windows variant vs. a macOS variant).

  4. 34. Define 'Release'.

    An instance of a system that is distributed to users outside of the development team.

  5. 35. What are the three basic techniques for component version identification?

    1. Version numbering. 2. Attribute-based identification. 3. Change-oriented identification.

  6. 36. What is a major drawback of simple version numbering (e.g., V1.1, V1.2)?

    The names are not meaningful, and the actual derivation structure is a tree or network (with branches and merges), not a simple linear sequence.

  7. 37. What is attribute-based identification?

    Associating a version with a combination of attributes (e.g., Date, Creator, Language, Status) that uniquely identify it.

  8. 38. What is the main advantage of attribute-based identification?

    It is very flexible and can support powerful queries, such as "find the most recent version for the XP platform written in Java."

  9. 39. What is change-oriented identification?

    A system that integrates versions with the specific changes ('change sets') that were applied to create them. This allows a version to be created by applying an arbitrary sequence of change sets.

  10. 40. What is the primary function of a Version Management tool regarding storage?

    Storage management. Modern systems typically store the differences (deltas) between versions rather than a full copy of every version, which saves a large amount of disk space.

  11. 41. How do version management tools support independent development?

    They manage check-outs and check-ins, often allowing only one person to modify a file at a time (locking) or by providing tools to merge parallel changes made to different versions (branching and merging).

Section 5: Release Management

  1. 42. What is the main concern of release management?

    Planning when to issue a system version as a formal release to customers.

  2. 43. What are the two main types of changes that a release must incorporate?

    1. Changes forced on the system (e.g., bug fixes, hardware changes). 2. New system functionality.

  3. 44. List three items, other than executable programs, that are typically included in a system release.

    1. Configuration files. 2. Data files needed for operation. 3. An installation program. (Also: documentation, packaging).

  4. 45. What is a potential problem with releasing a new version to a customer?

    The customer may not want a new release. They may be happy with their current system, and the new version might provide unwanted or disruptive functionality.

  5. 46. What should release management *not* assume about previous releases?

    It should not assume that all previous releases have been accepted and installed by the customer. A new release should be a complete installation.

  6. 47. List three factors that influence release decision making.

    Technical quality of the system, competition, marketing requirements, and customer change requests.

  7. 48. What is the purpose of release creation?

    To collect all the necessary files and documentation, write configuration and installation scripts, and create the final package for distribution.

  8. 49. Why is it important to document exactly what files were used to create a specific release?

    So that the release can be perfectly re-created if necessary, for example, to investigate a bug reported in that specific version.

Section 6: System Building

  1. 50. What is 'system building'?

    The process of compiling and linking software components into a final executable system.

  2. 51. How is the system building process automated?

    It is supported by automated tools that are driven by 'build scripts', such as a Makefile.

  3. 52. Describe a common system building problem related to component versions.

    Specifying the wrong version of a component in the build script. The system may build and work initially, but fail after delivery due to an incompatibility.

  4. 53. Why is embedding absolute file paths in code a system building problem?

    Because naming conventions and directory structures differ from place to place (e.g., between a developer's machine and a production server), causing the build to fail.

  5. 54. How can different compiler versions cause system building problems?

    Different compiler versions may generate slightly different code, causing the compiled component to exhibit different, unexpected behavior, which can lead to subtle bugs.

  6. 55. What is the purpose of the 'clean' target in a Makefile?

    To remove all compiled object files and the final executable, allowing for a fresh build from scratch.

  7. 56. In the system building diagram, what is the role of the 'Version management system'?

    It provides the correct versions of the source code components to the compilers, as specified by the build script.

  8. 57. What is 'delta-based versioning'?

    A storage management technique where a version control system stores only the differences (deltas) between versions of a file, rather than a full copy of each version, to save space.

  9. 58. Why is building a large system computationally expensive?

    Because it can involve compiling and linking hundreds or thousands of files, a process which can take several hours on a powerful machine.

  10. 59. What is 'distributed compilation'?

    A feature of advanced system building tools that uses multiple machines on a network to compile different parts of the system in parallel, speeding up the build process.

  11. 60. What does a 'linker' do in the build process?

    It takes the compiled object code from all the different components and combines them into a single executable system.

Section 7: CASE Tools for Configuration Management

  1. 61. Why is CASE tool support essential for CM?

    Because CM processes are standardized and involve managing large amounts of data, making manual tracking impractical and error-prone. Tools are essential for automation and control.

  2. 62. What is the range of available CASE tools for CM?

    They range from stand-alone tools (like a version control system) to fully integrated CM workbenches.

  3. 63. List three features of a change management tool.

    1. A form editor for change request forms. 2. A workflow system to automate information transfer. 3. A change database to manage proposals.

  4. 64. List three features of a version management tool.

    1. Version and release identification. 2. Storage management (e.g., delta-based). 3. Change history recording.

  5. 65. What does a workflow system in a change management tool do?

    It defines who does what in the change process and automates the transfer of information, for example, by emailing a change request to the next person in the approval chain.

  6. 66. What is the benefit of a change database being linked to a version management system?

    It provides full traceability, allowing you to link a specific change proposal directly to the versions of the code files that were modified to implement it.

  7. 67. What is "project support" in the context of version management tools?

    The ability of the tool to manage groups of files associated with a project (or a release) as a single logical unit, rather than just managing individual files.

  8. 68. What is the purpose of dependency specification language in a system building tool?

    To allow developers to define the relationships between components, so the tool knows which components need to be recompiled when another component changes (e.g., the rules in a Makefile).

  9. 69. What is 'derived object management' in a system building tool?

    The ability of the tool to manage and clean up the intermediate files created during a build (like object files), ensuring that outdated objects are not accidentally linked into a new build.

  10. 70. What is the most common and fundamental CASE tool for CM?

    A Version Management System or Version Control System (VCS), such as Git, Subversion, or CVS.

  11. 71. What is the key point about system releases?

    System releases include not just executable code, but also data, configuration files, and documentation.

  12. 72. What is the key point about version identification?

    A consistent scheme should be established using version numbers, attributes, or change sets.

  13. 73. What is the key point about the configuration database?

    It should record information about changes and change requests.

  14. 74. What is the key point about document management in CM?

    A formal document naming scheme should be established, and documents should be managed in a database.

  15. 75. What is the overall definition of CM from the key points?

    Configuration management is the management of system change to software products.

  16. 76. Why is a hierarchical naming scheme like `PCL-TOOLS/EDIT/FORMS/...` considered flexible?

    It allows for the logical grouping of related components and makes it easy to understand the context and purpose of a file from its name alone.

  17. 77. What is the advantage of a version control system storing only the differences (deltas)?

    It significantly reduces the amount of storage space required to maintain the complete history of a project, especially for text-based files like source code.

  18. 78. How does a Makefile know which files to recompile?

    It checks the modification timestamps. If a source file (like `main.c`) is newer than its corresponding object file (`main.o`), it knows it needs to be recompiled.

  19. 79. What is the final output of the system building process?

    A single executable system.

  20. 80. What is the relationship between the CM Plan and the CM Database?

    The CM Plan *defines* the structure and purpose of the CM Database that will be used to record configuration information for the project.

  21. 81. What does "independent development" support in a version management tool prevent?

    It helps prevent conflicting changes, where two developers might unknowingly overwrite each other's work on the same file.

  22. 82. What problem does a change tracking tool solve?

    The major problem of tracking the status of numerous change requests, ensuring none are lost and that they are routed to the correct people in a timely manner.

  23. 83. Who typically makes up a Change Control Board (CCB)?

    It may include representatives from both the client and contractor staff to provide a balanced strategic view.

  24. 84. Why is release decision making described as an "expensive process"?

    Because it involves not just building the software, but also creating documentation, installation packages, marketing materials, and coordinating distribution and support.

  25. 85. What information is typically found in a component header for CM purposes?

    Project name, component name/path, author, creation date, and a modification history (version, modifier, date, change, reason).

  26. 86. Can a 'release' contain multiple 'versions' of different components?

    Yes. A system release is composed of a specific, defined set of versions of all its constituent components.

  27. 87. What is the main purpose of a 'build script'?

    To automate the entire system building process, ensuring it is repeatable, consistent, and less prone to human error.

  28. 88. Why is it important that a build not rely on "standard" data files?

    Because "standard" can vary from one machine or environment to another. All required data files should be explicitly included and managed as part of the configuration to ensure a consistent build.

  29. 89. How are version management and change management integrated?

    They are integrated by linking change requests from a change management system to the specific code versions in the version management system that implement those changes, providing full traceability.

  30. 90. What is the difference between a 'configuration item' and a 'component'?

    A 'component' usually refers to a part of the software (like a source file). A 'configuration item' is a broader term for *any* artifact under CM control, including documents, test plans, and manuals, in addition to components.

  31. 91. What is the goal of a hierarchical configuration scheme?

    To organize thousands of documents so that related items have related names, making the system structure easier to understand and navigate.

  32. 92. What is the fundamental problem that CM addresses in team-based development?

    It addresses the problem of multiple people working on and changing a complex system concurrently, ensuring that changes are controlled, integrated smoothly, and do not conflict.

  33. 93. Why must a CM plan define responsibilities?

    To ensure accountability. It clarifies who has the authority to approve changes, create baselines, and manage the CM process, preventing chaos.

  34. 94. What is the role of the 'linker' in system building?

    After individual components are compiled into object code, the linker combines these object files and resolves dependencies between them to create a single executable file.

  35. 95. What is the difference between an object file and an executable file?

    An object file is the result of compiling a single source code file. An executable file is the result of linking multiple object files together into a runnable program.

  36. 96. Can a 'variant' also have different 'versions'?

    Yes. For example, you could have Version 1.1 and Version 1.2 of the Windows variant of your software.

  37. 97. What is the primary advantage of storing change history?

    It provides an audit trail, allowing developers to understand *why* a change was made, which is crucial for future maintenance and debugging.

  38. 98. How do CASE tools for CM support concurrent development?

    Version control systems, a key CASE tool, provide features like branching and merging, which allow developers to work in parallel on different features and then integrate their changes in a controlled manner.

  39. 99. What is the relationship between the 'build script' and the 'system builder' tool?

    The 'build script' (e.g., Makefile) contains the instructions, and the 'system builder' (e.g., the `make` utility) is the tool that reads and executes those instructions to build the system.

  40. 100. What is the overall message of the "Configuration Management" presentation?

    That managing change in software development is a complex but essential activity that requires a planned, standardized, and tool-supported process covering change control, versioning, releasing, and system building to ensure the integrity and quality of an evolving software product.

Part 6: Software Project Management

Section 1: Fundamentals of Software Project Management

  1. 1. What is the primary concern of software project management?

    Ensuring that software is delivered on time, on schedule, and in accordance with the requirements of the organization.

  2. 2. Why is project management particularly needed for software development?

    Because software development is always subject to budget and schedule constraints set by the developing organization.

  3. 3. List three key 'distinctions' or challenges that make software management different from other types of engineering management.

    1. The product is intangible. 2. The product is uniquely flexible. 3. The development process is not standardized.

  4. 4. Explain what it means for the software product to be 'intangible'.

    Unlike a physical product like a bridge, you cannot see or touch the software product itself. Progress is difficult to visualize and measure without formal mechanisms.

  5. 5. Explain what it means for the software product to be 'uniquely flexible'.

    It is usually possible to change software at any stage of its development, which means requirements are often not stable, making planning difficult.

  6. 6. What does it mean that many software projects are 'one-off' projects?

    Each project is often new and different from previous projects, so it is hard to reuse experience and established processes.

  7. 7. Despite its distinctions, what 'commonalities' does software management share with other engineering disciplines?

    Many general project management techniques are equally applicable, and technically complex systems in any field tend to suffer from similar problems (e.g., complexity, changing requirements).

  8. 8. Who sets the budget and schedule constraints for a software project?

    The organization developing the software.

Section 2: Management Activities

  1. 9. List six fundamental management activities.

    1. Proposal writing. 2. Project planning and scheduling. 3. Project costing. 4. Project monitoring and reviews. 5. Personnel selection and evaluation. 6. Report writing and presentations.

  2. 10. What is the purpose of 'proposal writing'?

    To describe the objectives of a project and how it will be carried out, often as part of a bid to win a contract from an external customer.

  3. 11. What is 'project costing'?

    The activity of predicting the resources (time, effort, materials) required to carry out the project.

  4. 12. What does 'project monitoring and reviews' involve?

    Continuously checking the progress of the project against its plan and reviewing the quality and standards of the software being developed.

  5. 13. What is the main responsibility of a project manager regarding report writing?

    To report on the project's progress to both the client and their own contractor organization.

  6. 14. What skills must a manager have for presentations?

    They must be able to present critical project information clearly and concisely during progress reviews.

  7. 15. What is 'personnel selection and evaluation'?

    The activity of choosing the right people for the project team and evaluating their performance.

  8. 16. List three reasons why it might not be possible to appoint the 'ideal' people to a project.

    1. The project budget may not allow for highly-paid staff. 2. Staff with the appropriate experience may not be available. 3. The organization may wish to use the project to develop the skills of junior employees.

  9. 17. What is the most time-consuming project management activity?

    Project planning.

  1. 18. Why is project planning a continuous activity?

    Because plans must be regularly revised as new information becomes available and as the project progresses. It is not a one-time, upfront activity.

  2. 19. What is the purpose of the main software project plan?

    It is concerned with the overall schedule and budget of the project.

    The manager draws up a schedule, initiates activities, reviews progress, revises estimates, and updates the schedule. This loop continues until the project is complete.

  6. 23. What three things does the main project plan set out?

    1. The resources available to the project. 2. The work breakdown. 3. A schedule for the work.

  7. 24. List four key sections of a project plan structure.

  10. 27. Give an example of a milestone in the requirements engineering process.

    The completion of the 'Feasibility report' or the 'Architectural design' document.

Section 5: Risk Management

  1. 42. What is 'risk management'?

    The process of identifying risks and drawing up plans to minimize their effect on a project.

  2. 43. What is a 'risk'?

    A probability that some adverse circumstance will occur.

  3. 44. List the three categories of risks.

    1. Project risks. 2. Product risks. 3. Business risks.

  4. 45. What do 'project risks' affect?

    The project schedule or resources. Example: Staff turnover, hardware unavailability.

  5. 46. What do 'product risks' affect?

    The quality or performance of the software being developed. Example: Requirements change, size underestimate.

  6. 47. What do 'business risks' affect?

    The organization developing or procuring the software. Example: Technology change, a competitor releasing a product first.

  7. 48. List the four stages of the risk management process.

    1. Risk identification. 2. Risk analysis. 3. Risk planning. 4. Risk monitoring.

  8. 49. What is 'risk identification'?

    Identifying potential project, product, and business risks.

  9. 50. What is 'risk analysis'?

    Assessing the likelihood (probability) and consequences (seriousness) of each risk.

  10. 51. What is 'risk planning'?

    Drawing up plans and strategies to avoid or minimize the effects of the risk.

  11. 52. What is 'risk monitoring'?

    Regularly assessing the identified risks throughout the project to see if their probability or impact has changed.

  12. 53. List three types of risks identified during risk identification.

    Technology risks, People risks, Organisational risks, Requirements risks, Estimation risks.

  13. 54. Give an example of a 'People risk'.

    Key staff are ill and unavailable at critical times; it is impossible to recruit staff with the required skills.

  14. 55. Give an example of an 'Estimation risk'.

    The time required to develop the software is underestimated; the size of the software is underestimated.

  15. 56. List the possible levels of risk probability.

    Very low, low, moderate, high, or very high.

  16. 57. List the possible levels of risk effect.

    Catastrophic, serious, tolerable, or insignificant.

  17. 58. What are the three types of strategies developed during risk planning?

    1. Avoidance strategies. 2. Minimisation strategies. 3. Contingency plans.

  18. 59. What is an 'avoidance strategy'?

    A strategy where the probability that the risk will arise is reduced (e.g., using a more reliable technology).

  19. 60. What is a 'minimisation strategy'?

    A strategy where the impact of the risk on the project or product will be reduced if it occurs (e.g., reorganizing the team so there is more knowledge overlap).

  20. 61. What is a 'contingency plan'?

    A plan to deal with a risk *if* it arises. It is a backup plan.

  21. 62. What is a 'risk indicator'?

    A potential indicator or warning sign that a particular type of risk may be materializing.

  22. 63. Give an example of a potential indicator for a 'People risk'.

  23. 64. Give an example of a potential indicator for an 'Estimation risk'.

    Failure to meet the agreed schedule or failure to clear reported defects at the expected rate.

  24. 65. What is a key point about the intangible nature of software?

    The intangible nature of software causes significant problems for management in tracking progress and quality.

  25. 66. What are the most significant activities of a software project manager?

Part 7: Software Evolution

Section 1: Evolution Fundamentals

  1. 1. Why is software change considered inevitable?

    Because new requirements emerge as software is used, the business environment changes, errors must be repaired, and the system must adapt to new hardware and technology to remain useful.

  2. 2. What is the key problem for organizations regarding their existing software?

    Implementing and managing change to their existing software systems in a controlled and cost-effective way.

  3. 3. How does the spiral model represent development and evolution?

    It shows development and evolution as a continuous spiral process, where initial development leads to Release 1, which then undergoes further specification, implementation, and validation to become Release 2, and so on.

  4. 4. What is 'program evolution dynamics'?

    It is the study of the processes of system change.

Section 2: Lehman's Laws of Software Evolution

  1. 5. What did Lehman and Belady propose after their empirical studies?

    They proposed a number of 'laws' (better described as observations) which applied to all large systems as they evolved.

  2. 6. Explain Lehman's Law of 'Continuing Change'.

    A program used in a real-world environment must change, or it will become progressively less useful.

  3. 7. Explain Lehman's Law of 'Increasing Complexity'.

    As an evolving program changes, its structure tends to become more complex. Extra resources must be devoted to preserving and simplifying the structure (i.e., refactoring).

  4. 8. Explain Lehman's Law of 'Large Program Evolution'.

    Program evolution is a self-regulating process. System attributes like size and time between releases are approximately invariant for each release.

  5. 9. Explain Lehman's Law of 'Organisational Stability'.

    Over a program's lifetime, its rate of development is approximately constant and independent of the resources devoted to it.

  6. 10. Explain Lehman's Law of 'Conservation of Familiarity'.

    The incremental change in each release is approximately constant, as users cannot cope with too much change at once.

  7. 11. Explain Lehman's Law of 'Continuing Growth'.

    The functionality offered by systems has to continually increase to maintain user satisfaction.

  8. 12. Explain Lehman's Law of 'Declining Quality'.

    The quality of systems will appear to be declining unless they are actively adapted to changes in their operational environment.

  9. 13. Explain Lehman's Law of 'Feedback System'.

    Evolution processes must be treated as multi-loop feedback systems to achieve significant product improvement.

  10. 14. To what types of systems are Lehman's laws most applicable?

    To large, tailored (bespoke) systems developed by large organizations.

  11. 15. For what types of software is it unclear how Lehman's laws should be modified?

    Shrink-wrapped software products, systems with many COTS components, and systems developed by small organizations.

Section 3: Software Maintenance

  1. 16. What is 'software maintenance'?

    Modifying a program after it has been put into use.

  2. 17. Does maintenance normally involve major architectural changes?

    No, maintenance does not normally involve major changes to the system's architecture. Changes are implemented by modifying existing components and adding new ones.

  3. 18. List the four types of scheduled maintenance activities.

    Adaptive, Preventive, Perfective, and Corrective.

  4. 19. What is 'Adaptive maintenance'?

    Adapting the software to a new operating environment, new hardware, or new technology.

  5. 20. What is 'Corrective maintenance'?

    Diagnosing and making changes to fix software errors (bugs).

  6. 21. What is 'Perfective maintenance'?

    Making changes to improve both functional and non-functional requirements, such as enhancing performance or usability.

  7. 22. What is 'Preventive maintenance'?

    Making software changes to improve future maintainability and reduce the future discovery of errors. This includes activities like refactoring and re-engineering.

  8. 23. What is 'emergency maintenance'?

    Unscheduled corrective maintenance to deal with a critical error that must be dealt with immediately.

  9. 24. What is the approximate distribution of maintenance effort according to the lecture?

    75% are adaptive and preventive; 20% corrective, 5% perfective.

  10. 25. How does 'team stability' affect maintenance costs?

    Maintenance costs are reduced if the same staff who developed the system also maintain it, as they have domain knowledge and understand the code.

  11. 26. How does 'contractual responsibility' affect maintenance?

    If the original developers have no contractual responsibility for maintenance, they have no incentive to design the system for future change, increasing later maintenance costs.

  12. 27. How do 'staff skills' typically affect maintenance costs?

    Maintenance staff are often less experienced and have limited domain knowledge, which can make understanding and changing the code more difficult and costly.

  13. 28. How do 'program age and structure' affect maintenance?

    As programs age, their structure tends to degrade (see Lehman's Law of Increasing Complexity), and they become harder to understand and change.

  14. 29. What is 'maintenance prediction' concerned with?

    Assessing which parts of a system may cause problems and have high maintenance costs in the future.

  15. 30. What is the relationship between maintainability and change acceptance?

    The decision to accept a proposed change depends on the maintainability of the components affected by that change. If the code is easy to change, the cost is lower and the change is more likely to be accepted.

Section 4: The Evolution Process

  1. 31. What is the main driver for system evolution?

    Proposals for change (change requests).

  2. 32. What three factors do evolution processes depend on?

    1. The type of software being maintained. 2. The development processes used. 3. The skills and experience of the people involved.

  3. 33. Describe the cycle of change identification and evolution.

    A new system is created, which leads to a change identification process that generates change proposals. These proposals feed into the software evolution process, which results in a new version of the system, and the cycle repeats.

  4. 34. List the five main stages of the system evolution process.

    1. Change Requests -> 2. Impact Analysis -> 3. Release Planning -> 4. Change Implementation -> 5. System Release.

  5. 35. What is 'impact analysis'?

    The process of assessing the effect of a proposed change on the rest of the system, including estimating the cost and effort required to implement it.

  6. 36. What is 'release planning'?

    The process of deciding which validated change requests will be implemented in the next release of the system.

  7. 37. What are the three outputs of the release planning stage?

    Fault repair, Platform adaptation, and System enhancement.

  8. 38. List the four stages of the 'change implementation' sub-process.

    Proposed changes -> Requirements analysis -> Requirements updating -> Software development.

  9. 39. List three situations that might require an urgent change request.

    1. A serious system fault has to be repaired. 2. A change to the system's environment (e.g., an OS upgrade) has unexpected effects. 3. A business change requires a very rapid response (e.g., a competitor's release).

  10. 40. How does the process for an urgent change (emergency repair) differ from the standard process?

    It may have to be implemented without going through all the stages of the software engineering process. The process is often shortened to: Analyze source code -> Modify source code -> Deliver modified system.

    It has reduced risk compared to new software development because the requirements of the system are already well-understood.
